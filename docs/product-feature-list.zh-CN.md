# 产品视角功能清单

本文档从产品、运营和业务视角总结当前项目已经具备的能力，适合作为以下场景的快速材料：

- 向产品经理解释“这个系统现在能做什么”
- 向运营或商务解释“这个系统能怎么卖、怎么管”
- 向管理层解释“这不是单一 API 转发器，而是一套 AI 平台”

如需查看代码视角的拆解，请同时参考：

- `docs/project-feature-overview.zh-CN.md`

## 1. 一句话定义

这是一个面向 AI 模型接入、统一分发、计费结算、用户管理和后台运营的综合平台。

如果从业务上描述，它更接近：

- AI 模型聚合平台
- 面向团队或客户的 API 服务平台
- 带钱包和订阅体系的 AI 网关后台
- 可运营、可计费、可审计的 AI 中台

## 2. 它解决的核心问题

如果不使用这套系统，团队通常要分别处理这些问题：

- 对接多个模型厂商，协议不统一
- 不同模型 key 分散管理，无法统一审计
- 不知道如何给用户分配额度和收费
- 缺少用户登录、注册、充值、订阅能力
- 缺少后台查看消耗、日志、渠道状态
- 缺少对失败重试、限流、风控、监控的统一控制

这个项目的价值就在于把这些问题合成到一个系统里。

## 3. 适合的使用对象

### 3.1 平台管理员

平台管理员可以用它来：

- 接入和管理多个上游模型供应商
- 给不同用户、分组、Token 配置访问权限
- 统一做充值、订阅、定价和额度分发
- 查看用户使用情况和调用日志
- 做风控、监控、限流和系统配置

### 3.2 内部团队用户

内部研发、运营或业务团队可以用它来：

- 统一调用不同模型接口
- 在 Playground 或聊天页里直接使用模型
- 管理自己的 Token 和额度
- 查看调用消耗、日志、绘图记录、任务记录

### 3.3 外部客户或租户

如果作为对外平台运营，外部客户可以：

- 注册账号并登录平台
- 充值或订阅购买额度
- 创建和管理 API Token
- 使用统一接口访问多个 AI 模型
- 在前台查看余额、账单和调用情况

## 4. 产品能力地图

### 4.1 AI 网关能力

这是平台最底层的能力，核心价值是“统一接入和调用”。

业务上可以理解为：

- 给外部或内部用户提供统一的 AI API
- 屏蔽不同模型厂商之间的协议差异
- 让调用方尽量只面对一种接入方式

当前已覆盖的主要能力有：

- OpenAI 兼容接口
- OpenAI Responses
- Realtime
- Claude Messages
- Gemini
- Embedding
- Audio
- Image
- Rerank
- Midjourney
- Suno

对业务的意义：

- 更容易快速上线多模型能力
- 降低接入多个厂商的开发成本
- 后续更换上游时对客户影响较小

### 4.2 多模型与多供应商聚合

平台并不是绑定单一供应商，而是允许把多个模型渠道统一放在后台管理。

这意味着它可以支持如下业务模式：

- 一个平台售卖多个模型能力
- 同一模型接多个上游渠道做容灾
- 同时提供国外模型和国内模型
- 混合提供文本、绘图、语音、视频等能力

这部分决定了它适合做“模型超市”或“统一 AI 服务入口”。

### 4.3 用户与身份体系

从产品上看，这个平台已经不是“只有管理员能用”的工具，而是有完整账户体系的产品。

支持的能力包括：

- 注册、登录、找回密码
- 邮箱验证
- 2FA
- Passkey
- 第三方 OAuth 登录与绑定

业务意义：

- 能做正式用户系统
- 能做更安全的企业或会员平台
- 便于后续接入邀请、推广、分层用户体系

### 4.4 钱包与额度体系

这是平台商业化的基础。

从产品角度，它已经具备：

- 钱包页
- 充值
- 兑换码
- 邀请返利
- 额度转赠
- 额度展示与消耗跟踪

这意味着平台可以同时支持：

- 自营内部使用
- 预付费模式
- 活动发码模式
- 邀请裂变模式

### 4.5 订阅能力

除了单次充值，平台还支持订阅计划和订阅购买。

从业务模式上，这很重要，因为它让平台不仅能卖“余额”，还可以卖：

- 月度套餐
- 年度套餐
- 特定用户包
- 特定分组包
- 订阅优先扣费模式

这决定了平台更适合长期经营，而不是一次性交付。

### 4.6 Token 产品化能力

平台把 API Token 做成了正式产品对象，而不是简单字段。

用户可以：

- 创建 Token
- 查询 Token
- 删除或更新 Token
- 获取 Token key
- 查看 Token 用量

业务意义：

- 支持开发者平台模式
- 支持个人或团队多 Token 管理
- 支持按 Token 做权限和配额控制

### 4.7 管理后台能力

管理员后台不是只有“配置 key”，而是一整套运营后台。

当前后台包含的核心页面包括：

- 渠道管理
- 模型管理
- 模型部署
- 订阅管理
- 用户管理
- 兑换码管理
- 系统设置

这意味着产品已经具备“后台产品”形态，而不是单纯技术服务。

### 4.8 数据看板与日志能力

平台已经内置可视化和审计能力，主要包括：

- 数据看板
- 消耗统计
- 使用日志
- Midjourney 日志
- 任务日志
- Uptime 状态
- API 信息面板

业务意义：

- 管理员能看平台运行状况
- 用户能看自己的使用情况
- 支持售后、审计、问题排查

### 4.9 风控与运营配置

从产品成熟度看，这部分很关键。当前系统已暴露大量运营配置：

- 敏感词过滤
- 自动禁用异常渠道
- 自动检测渠道状态
- 日志策略
- 签到送额度
- 顶栏和侧边栏模块开关
- Token 数量限制
- 新用户额度配置
- 邀请奖励配置

业务意义：

- 平台可以做活动和运营策略
- 平台可以控制风险和成本
- 平台可以按阶段逐步开放功能

### 4.10 支付能力

目前已经具备多支付通道接入能力，包括：

- EPay
- Stripe
- Creem
- Waffo

这意味着产品在商业化上不是概念验证，而是已经具备支付闭环的基础条件。

### 4.11 模型部署能力

这是一个比普通 API 网关更进一层的能力。

当前项目不只调用模型，还开始具备：

- 部署模型工作负载
- 估算部署成本
- 选择硬件与地域
- 管理部署生命周期
- 查看部署日志和容器信息

从产品角度，这意味着未来可以往两个方向发展：

- “模型调用平台”
- “模型调用 + 模型部署”的一体化平台

## 5. 典型业务场景

### 场景 1：企业内部统一 AI 网关

适合做法：

- 管理员接入多个模型供应商
- 内部员工统一通过平台调用
- 用分组、Token 和额度限制不同团队权限
- 用日志和统计看部门使用情况

价值：

- 降低多模型接入复杂度
- 统一安全、审计和费用控制

### 场景 2：对外售卖 AI API

适合做法：

- 开放注册登录
- 用户充值或购买订阅
- 用户创建 Token 调用统一 API
- 平台后台管理渠道、模型、定价和风险

价值：

- 快速搭建 API 商业化平台
- 支持预付费和订阅双模式

### 场景 3：面向社区或会员的 AI 服务站

适合做法：

- 开启聊天页、Playground、模型广场
- 通过邀请码、兑换码、签到送额度拉新
- 用 OAuth 降低注册门槛

价值：

- 更偏产品化体验，而不是纯开发者工具

### 场景 4：混合模型运营平台

适合做法：

- 同时提供文本、绘图、音频、任务类能力
- 用渠道管理把不同厂商能力统一到一个平台
- 针对不同模型单独定价

价值：

- 更容易形成多品类 AI 服务矩阵

## 6. 当前已经具备的商业化方式

从现有代码能力推断，当前系统已经支持的商业化方式包括：

- 按充值余额消费
- 按订阅套餐消费
- 按模型或分组差异化定价
- 邀请奖励和推广返利
- 兑换码活动投放

这意味着它已经具备从“能用”走向“能运营、能收费”的必要条件。

## 8. 当前产品边界

虽然平台能力已经较完整，但它目前仍有一些边界需要明确：

- 它已经是综合 AI 平台，但并非所有 OpenAI 接口都已完全覆盖
- 一些接口仍是未实现状态，例如 files、fine-tunes、images variations 等
- 模型部署能力已经存在，但仍更像平台扩展能力，不是当前唯一主线

因此更准确的产品描述应是：

“一个以 AI 模型聚合、统一网关、计费管理和平台运营为核心的综合系统。”

而不是：

- “纯 OpenAI 替代品”
- “单一聊天应用”
- “只有管理员用的技术后台”

## 9. 对外介绍时的推荐说法

如果要对外介绍这个项目，建议优先使用以下表达：

### 面向技术团队

这是一个支持多模型、多供应商、多协议兼容的 AI 网关平台，提供统一 Relay、用户体系、Token 管理、计费和后台运营能力。

### 面向产品或老板

这是一个可以直接支撑 AI 平台运营的底座，既能统一接入模型，也能管理用户、收费、做风控和看数据。

### 面向客户或合作方

这是一个把多家 AI 能力整合到统一接口与统一控制台中的平台，支持充值、订阅、权限控制和调用审计。

## 10. 结论

从产品视角看，这个项目已经具备以下关键特征：

- 不是单点工具，而是平台型产品
- 不是只做代理，还覆盖用户、计费、后台和运营
- 不是只面向开发者，也面向平台用户
- 具备较明确的商业化与运营潜力

如果继续往产品化方向推进，后续最值得补强的通常会是：

- 更清晰的租户/组织能力
- 更完整的账单和财务视图
- 更完善的客服与审计工具
- 更强的模型市场和套餐包装能力
