# 16-结算、佣金、退款与冲销公式设计

更新时间：2026-04-16

## 1. 文档目标

本文件冻结 Marketplace 中“支付成功后的平台代收资金”如何转化为“卖家可结算收入”的口径，重点解决：

1. 卖家收入是在支付时确认，还是在履约时确认
2. 平台抽佣、卖家净收入、退款冲销如何计算
3. entitlement 未消耗到期时，未履约义务如何释放
4. 如何保证 `order / entitlement / usage / settlement` 四条链可对账
5. 如何避免直接修改历史事实记录

## 2. 适用边界

- `M1` 只支持 `text/chat`、按 `token` 计量的 entitlement
- 支付成功后平台先形成代收资金，不立即释放卖家可结算余额
- `M1` 可先落 `settlement_entry` 事实账，不要求做完整提现释放
- `M2` 再实现 `releasable -> settled` 的财务出账流程
- 所有金额统一用 `amount_minor`，不使用 `float64`

## 3. 冻结决策

### 决策 A：卖家收入按“履约事件”确认，不按支付事件确认

- 支付成功只代表平台代收成功
- 卖家收入确认事件只有两类：
  - 买方真实消耗 usage
  - 买方未消耗 entitlement 到期失效

### 决策 B：`settlement_entry` 是最小结算事实单元

- 每一条 entry 必须绑定唯一 source event
- 不允许事后根据余额反推结算
- 历史 entry 不允许覆盖更新金额，只允许追加冲销 entry

### 决策 C：平台抽佣口径冻结在订单项快照，不跟随后续配置漂移

- `platform_fee_bps`
- `guarantee_days`
- 其他营销减免口径

以上值都应在订单项或可追溯快照中冻结，后续配置修改不反向影响已成交订单。

### 决策 D：退款先释放未履约义务，再冲销已确认收入

- 未使用部分退款：优先释放 `ReservedCapacity` 与平台代收义务
- 已确认收入部分退款：必须生成 reversal entry
- 不允许直接回写旧 settlement entry 的 `gross/net`

### 决策 E：尾差只允许在单订单项内收口

由于 `line_amount_minor / granted_amount` 可能除不尽：

- 非最后一次结算按公式向下取整
- 同一订单项最后一笔结算或到期释放时吃掉剩余尾差

这样可保证：

- 单订单项所有正向 entry 的 `gross_amount_minor` 总和
- 等于该订单项最终未退款的可分配金额

## 4. 核心概念

### 4.1 资金层

- `cash_collected_minor`
  - 平台实际收款
- `cash_refunded_minor`
  - 平台已退款金额

### 4.2 履约层

- `granted_amount`
  - 已发放 entitlement 总量
- `used_amount`
  - 已真实消耗量
- `expired_amount`
  - 到期失效量
- `refunded_amount`
  - 已退款量

### 4.3 结算层

- `gross_amount_minor`
  - 本次履约对应毛收入
- `platform_fee_minor`
  - 本次平台抽佣
- `net_amount_minor`
  - 本次卖家净收入

## 5. 公式冻结

## 5.1 单订单项可分配金额

对单个 `market_order_item`：

```text
AllocatableAmountMinor = LineAmountMinor - RefundedCashMinor
```

说明：

- `LineAmountMinor` 来自订单项成交快照
- 若订单级存在优惠，应在下单时按明确规则分摊到订单项，再冻结到订单项

## 5.2 单位额度对应收入

概念公式：

```text
QuotaUnitPriceMinor = LineAmountMinor / GrantedAmount
```

实现规则：

- 不直接存浮点单价
- 每次结算使用整数公式计算

## 5.3 usage 结算毛收入

对某次 usage，若其结算额度为 `quota_delta`：

```text
GrossAmountMinor = floor(quota_delta * LineAmountMinor / GrantedAmount)
```

若该 usage 是该订单项最后一笔正向结算事件，则：

```text
GrossAmountMinor = RemainingAllocatableMinor
```

其中：

```text
RemainingAllocatableMinor =
  LineAmountMinor
  - RefundedCashMinor
  - Sum(PreviousPositiveSettlementGrossMinor)
```

## 5.4 entitlement 到期结算毛收入

对未消耗但已到期失效的额度 `expired_quota`：

```text
ExpiryGrossAmountMinor = RemainingAllocatableMinor
```

前提：

- 该部分 entitlement 未退款
- 已超过有效期
- 无待决争议冻结

作用：

- 结束平台对买方的未履约义务
- 把未消耗但已自然失效的商品价值转为卖家可结算收入

## 5.5 平台抽佣

```text
PlatformFeeMinor = floor(GrossAmountMinor * PlatformFeeBps / 10000)
NetAmountMinor = GrossAmountMinor - PlatformFeeMinor
```

佣金来源：

- 优先取订单项快照中的 `platform_fee_bps`
- 若 `M1` 尚未建快照字段，至少要从 listing/seller 配置在支付成功时冻结后写入可追溯字段

## 6. 结算事件模型

## 6.1 支付成功

支付成功时：

- 创建 entitlement
- `supply_account.reserved_capacity += granted_amount`
- 不创建卖家可结算 entry

说明：

- 此时平台只形成“代收资金 + 履约义务”
- 卖家收入尚未确认

## 6.2 usage 成功

当请求成功并拿到最终 usage：

- `reserved_capacity -= actual_quota`
- `used_capacity += actual_quota`
- 创建一条 `source_type = usage` 的 `settlement_entry`
- `status = frozen`

## 6.3 entitlement 到期

当 lot 到期且存在未消耗未退款余额：

- `reserved_capacity -= expired_quota`
- lot 进入 `expired`
- 创建一条 `source_type = expiry_release` 的 `settlement_entry`
- `status = frozen`

## 6.4 支付后全额退款但未发生 usage

- 释放 `reserved_capacity`
- 不创建正向 settlement entry
- 平台现金直接回退买方

## 6.5 已部分 usage 后的部分退款

- 未使用部分：释放 `reserved_capacity`，不产生正向结算
- 已使用部分：保留已产生的 usage settlement
- 若平台决定对已结算部分补偿买方，必须新增 reversal entry

## 6.6 争议赔付 / 平台补偿

如果已确认收入部分被判定需要赔付：

- 原正向 entry 不改写
- 新增 `source_type = dispute_reversal` 或 `manual_compensation_reversal`
- 金额为负值或通过 `status = reversed` + 独立 reversal 字段表达

推荐做法：

- 保留原 entry 正数金额
- 追加一条反向 entry，便于审计和对账

## 7. `settlement_entry` 字段补充建议

在 `08` 既有字段基础上，建议增加或在 `Meta` 中冻结：

- `SourceType`
  - `usage/expiry_release/dispute_reversal/manual_adjustment`
- `SourceEventKey`
- `PlatformFeeBpsSnapshot`
- `GrantedAmountSnapshot`
- `SettledQuota`
- `Meta`
  - `request_id`
  - `entitlement_lot_id`
  - `refund_case_id`
  - `dispute_case_id`

## 8. 冻结期规则

建议：

```text
FreezeUntil = max(PaidAt, SourceEventFinishedAt) + GuaranteeDays * 86400
```

规则：

- 正向收入先进入 `frozen`
- 超过保障期且无争议，进入 `releasable`
- 财务执行出账后进入 `settled`
- 有争议时可进入 `withheld`

## 9. 幂等键建议

| 事件 | 幂等键 |
| --- | --- |
| usage 结算 | `usage_event_key + settlement_stage` |
| entitlement 到期释放 | `order_item_id + entitlement_lot_id + expiry_release_stage` |
| 退款冲销 | `refund_event_key + reversal_stage` |
| 争议冲销 | `dispute_case_id + reversal_stage` |

## 10. 与库存、退款、争议的联动

必须与以下文档一致：

- `12-支付订单与权益发放时序设计.md`
- `14-库存口径、超卖控制与自动停售规则设计.md`
- `15-Token与Entitlement绑定及资金源优先级设计.md`
- `17-RBAC、敏感操作与审计权限矩阵设计.md`

关键联动：

- 支付成功只增加 `ReservedCapacity`
- usage / expiry 才释放履约义务并确认卖家收入
- refund 优先减少未履约义务
- dispute 冲销通过 reversal entry 完成

## 11. 不建议的实现

- 支付成功即把整单金额记给卖家
- 用 `float64` 计算每 token 单价
- 直接修改旧 settlement entry 的金额字段
- entitlement 到期后不释放 `ReservedCapacity`
- 把退款直接理解为“删除原收入”

## 12. 验收标准

满足以下条件时，本专题视为完成：

1. 已冻结“支付收款”和“卖家收入确认”之间的边界
2. usage、expiry、refund、dispute 四类结算事件都有明确口径
3. 单订单项内可通过尾差规则做到账平
4. entitlement 到期后 `ReservedCapacity` 能被释放
5. 文档足以直接指导 `settlement_entry` 与后续 M2 清结算实现
