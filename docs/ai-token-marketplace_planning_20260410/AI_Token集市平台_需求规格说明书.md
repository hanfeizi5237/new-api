# AI Token集市平台 - 完整需求规格说明书

## 1. 项目概述

### 1.1 项目名称
**AI Token集市平台** (AI Token Marketplace)

### 1.2 项目愿景
建立一个开放、透明、安全的AI API调用额度（Token）二级交易市场，连接AI API额度供应方与需求方，通过市场化机制优化资源配置，降低AI使用门槛，打造去中心化的AI资源交易生态系统。

### 1.3 核心定位
- **平台性质**：技术赋能的中介平台，非服务提供方
- **交易标的**：AI模型API调用额度（Token）的使用权
- **核心价值**：
  - 对卖方：闲置API额度变现渠道
  - 对买方：灵活获取多源AI能力，降低使用成本
  - 对平台：构建AI资源交易生态，收取服务费

### 1.4 关键类比
- **交易模式**：类似"闲鱼"的C2C交易 + "藏宝阁"的虚拟商品交易
- **技术模式**：类似"API网关"的路由代理 + "负载均衡"的智能调度
- **市场模式**：类似"股票市场"的自由定价 + "保险市场"的风险分级

## 3. 核心业务流程

### 3.1 商品上架流程
```
卖方发起上架 → 填写商品信息 → 密钥实时验证 → 选择保障等级 → 缴纳保证金 → 商品审核 → 上架成功
```

### 3.2 额度购买流程
```
买方浏览商品 → 选择商品/数量 → 风险提示确认 → 支付 → 额度到账 → 评价（可选）
```

### 3.3 API调用流程
```
买方应用 → 调用平台统一API → 平台鉴权 → 路由决策 → 选择可用密钥 → 转发请求 → 返回结果 → 记录日志 → 扣除额度
```

### 3.4 争议处理流程
```
买方发起争议 → 提交证据 → 卖方响应 → 平台仲裁 → 裁决执行 → 信用记录更新
```

### 4.3 代理路由系统（核心技术）

**4.3.1 统一API网关**
```
接口规范：
- 请求端点：https://api.token-market.com/v1/{provider}/{model}
- 认证方式：Bearer Token（平台分配的Client Key）
- 请求格式：与官方API 100%兼容
- 响应格式：与官方API 100%兼容
- 流式支持：完整SSE支持
```

**4.3.2 路由策略引擎**
```yaml
路由策略配置：
- 默认策略：健康检查 + 响应时间优先
- 可选策略：
  - 优先级策略：按卖家信用/价格排序
  - 负载均衡：轮询或按余额权重
  - 成本优先：优先使用单价最低的
  - 性能优先：优先使用响应最快的
  - 自定义规则：基于HTTP头或参数的路由
```

**4.3.3 故障转移机制**
```
故障检测：
1. 健康检查：每分钟检查密钥状态
2. 实时监控：请求失败率、响应时间
3. 自动隔离：连续失败3次，隔离5分钟
4. 自动恢复：隔离期后重试，成功则恢复

切换流程：
1. 当前密钥失败
2. 标记为不可用
3. 从可用池选择下一个
4. 重试请求（最多3次）
5. 记录切换日志
```

**4.3.4 监控与日志**
- 请求日志：时间、用户、密钥、模型、Token消耗、响应时间、状态码
- 性能指标：QPS、成功率、P99延迟
- 告警规则：失败率>5%、响应时间>10s

### 4.4 计费结算系统

**4.4.1 实时扣费**
# 扣费逻辑伪代码
def deduct_tokens(user_id, provider, model, tokens_used):
    # 1. 查询用户持有的该厂商的所有商品
    products = get_user_products(user_id, provider, model)

    # 2. 按购买时间顺序消耗（先买先用）
    for product in sorted(products, key=lambda x: x.purchase_time):
        if product.remaining_tokens >= tokens_used:
            product.remaining_tokens -= tokens_used
            tokens_used = 0
            break
        else:
            tokens_used -= product.remaining_tokens
            product.remaining_tokens = 0

    # 3. 记录扣费流水
    record_deduction(user_id, provider, model, tokens_used)
```

**4.4.2 结算规则**
```
结算周期：T+1（交易完成后次日结算）
结算条件：
- 额度已消耗
- 无未解决争议
- 超过7天保障期
结算金额 = 消耗额度 × 单价 - 平台佣金
```

### 4.5 风控信用系统

**4.5.1 信用评分模型**
```
卖家信用分 = 基础分(300) + 交易分(0~500) - 违规分(0~∞)
交易分因子：
- 成功交易额：每1000元+1分
- 好评率：90%以上+100分
- 响应速度：平均<5s+50分
违规分因子：
- 争议败诉：每次-50分
- 虚假商品：每次-200分
- 恶意行为：每次-500分
```

**4.5.2 风险等级**
| 信用分 | 等级 | 权益 | 限制 |
|-------|-----|-----|-----|
| 800+ | 钻石 | 优先展示、低佣金、高额度 | 无 |
| 600-799 | 黄金 | 正常权益 | 正常 |
| 300-599 | 白银 | 基础权益 | 交易额度限制 |
| 0-299 | 青铜 | 受限权益 | 高保证金、人工审核 |
| <0 | 黑名单 | 无 | 禁止交易 |

**4.5.3 异常检测规则**
- 交易异常：单价低于市场价30%、高频小额交易
- 行为异常：多账号操作、API调用模式异常
- 安全异常：异地登录、密码尝试频繁

### 4.6 争议仲裁系统

**4.6.1 争议类型**
1. **密钥无效争议**：买方购买后密钥无法使用
2. **额度不符争议**：实际额度与描述不符
3. **服务质量争议**：响应时间过长、失败率高
4. **其他争议**：未按约定提供服务

**4.6.2 处理流程**
```
发起争议 → 提交证据 → 对方回应 → 平台仲裁 → 执行裁决 → 信用记录
时限要求：
- 买方：发现问题7天内发起
- 卖方：收到争议24小时内回应
- 平台：双方提交证据后48小时内裁决
```

**4.6.3 证据要求**
- 必须提供：调用失败截图、时间戳、请求ID
- 建议提供：日志文件、网络抓包
- 平台提供：调用日志查询接口

**4.6.4 裁决标准**
| 争议类型 | 卖方责任 | 裁决标准 |
|---------|---------|---------|
| 密钥无效 | 是 | 全额退款+信誉扣分 |
| 部分额度无效 | 是 | 按比例退款 |
| 服务质量差 | 视情况 | 部分退款或补偿 |
| 买方误操作 | 否 | 争议驳回 |

### 6.2 核心服务设计

**6.2.1 路由代理服务**
仅限于参考，具体实现要结合后续详细设计；

```go
// 核心路由逻辑
type Router struct {
    strategies map[string]RoutingStrategy
    healthChecker *HealthChecker
    cache *Cache
}

func (r *Router) Route(req *Request, userID string) (*Response, error) {
    // 1. 鉴权
    if err := r.auth(req, userID); err != nil {
        return nil, err
    }

    // 2. 获取用户可用密钥
    keys := r.getUserKeys(userID, req.Provider, req.Model)

    // 3. 过滤健康密钥
    healthyKeys := r.filterHealthyKeys(keys)

    // 4. 选择策略
    strategy := r.getStrategy(userID, req.Provider)
    selectedKey := strategy.Select(healthyKeys, req)

    // 5. 转发请求
    resp, err := r.forward(req, selectedKey)

    // 6. 处理结果
    if err != nil {
        r.markUnhealthy(selectedKey)
        return r.retryWithOtherKeys(req, healthyKeys, selectedKey)
    }

    // 7. 记录使用量
    r.recordUsage(userID, selectedKey, req, resp)

    return resp, nil
}
```
