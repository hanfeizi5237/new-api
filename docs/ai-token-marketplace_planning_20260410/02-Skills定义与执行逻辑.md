# 02-Skills定义与执行逻辑

## 1. 本次工作使用的方法组合

本次不是单纯的代码阅读，而是按“需求拆解 + 底座评估 + 架构设计 + 项目规划”的方式执行。

为此采用了以下技能与角色分工：

| 名称 | 类型 | 本次用途 |
| --- | --- | --- |
| `brainstorming` | 过程技能 | 先拆需求、识别系统边界，再输出设计而不是直接下结论 |
| `architecture-designer` | 设计技能 | 形成总体架构、模块边界和 ADR |
| `dispatching-parallel-agents` | 并行技能 | 将三个候选底座拆给独立小队评估 |
| `task-planning-skill` | 规划技能 | 将方案落成带 Phase 的实施计划和风险清单 |

## 2. ClawTeam 团队组织方式

本次按三层组织工作：

### 2.1 主线程角色

主线程负责：

- 阅读需求规格说明书
- 提炼系统边界
- 对齐现有仓库上下文
- 综合各候选底座结论
- 输出统一方案文档

### 2.2 并行评估小队

建立了三个独立评估小队，每队只负责一个仓库：

1. `coai-main` 评估队
2. `new-api-main` 评估队
3. `litellm-main` 评估队

每个小队的输出要求一致：

- 项目定位
- 技术栈与架构特征
- 与目标平台的契合点
- 明显缺口
- 是否适合作为底座

### 2.3 汇总层

汇总层负责：

- 建立统一的比较维度
- 去掉 README 层面的营销信息
- 只保留可以落到源码和路由/模型层的证据
- 转换成“可实施决策”，而不是“感受型推荐”

## 3. 每个技能的实际执行逻辑

### 3.1 `brainstorming`

使用目的：

- 防止把问题误解成“选网关框架”
- 在进入方案前先明确系统真正包含哪些子系统

本次落地动作：

- 先读取需求规格说明书
- 抽出功能和非功能约束
- 明确该项目不是单一代理站
- 将问题重构为“底座 + 新领域模块”的组合设计

输出价值：

- 让后续所有比较都基于同一个需求模型

### 3.2 `architecture-designer`

使用目的：

- 避免只做仓库比较，不做实际系统设计

本次落地动作：

- 设计模块化单体架构
- 规划领域边界
- 输出关键 ADR
- 给出可演进的部署和拆分路线

输出价值：

- 让方案可以直接进入实施设计，而不是停留在选型层

### 3.3 `dispatching-parallel-agents`

使用目的：

- 三个候选仓库彼此独立，适合并行分析

本次落地动作：

- 为每个仓库分配一名独立探索角色
- 提供统一背景和输出模板
- 避免主线程在三个仓库之间来回切换

输出价值：

- 保证每个候选仓库都能被完整看一遍
- 降低遗漏和主观偏差

### 3.4 `task-planning-skill`

使用目的：

- 把“设计意见”变成“可以推进的项目计划”

本次落地动作：

- 创建规划文档目录
- 形成执行阶段、风险清单、质量门禁
- 明确团队配置、里程碑与验收标准

输出价值：

- 让方案具备可追踪性

## 4. 技能之间的调用顺序

本次采用顺序如下：

1. `brainstorming`
2. `dispatching-parallel-agents`
3. `architecture-designer`
4. `task-planning-skill`

原因：

- 如果不先做需求拆解，底座评估会偏题
- 如果不并行评估，三个仓库会拖慢主路径
- 如果不做架构设计，最后只能得到“选哪个仓库”
- 如果不做任务规划，最终方案无法直接开工

## 5. 数据流转方式

### 5.1 输入

输入来源有四类：

- 需求规格说明书
- 当前 `one-api-main` 仓上下文
- `coai-main` 源码与文档
- `new-api-main` 源码与文档
- `litellm-main` 源码与文档

### 5.2 中间产物

中间产物包括：

- 需求拆解清单
- 候选底座证据列表
- 比较维度与评分
- 架构草图
- ADR 列表
- Phase 规划

### 5.3 最终产物

最终产物为本目录下的规划文档集：

- 总览
- 决策分析
- 技能和团队执行逻辑
- 技术能力与基础设施评估
- 完整执行步骤
- 风险处理方案
- 质量校验清单

## 6. 为什么没有直接进入代码实施

虽然已有明确主底座推荐，但本次没有直接改业务代码，原因是：

1. 当前用户请求是“全面分析和设计”
2. 目标系统仍有关键边界需要先冻结
3. 市场、结算、争议等领域如果边界没先定，直接编码会导致大量返工

因此本次交付聚焦于：

- 形成实施决策
- 把复杂问题拆成阶段
- 为下一轮正式开发铺路

## 7. 后续执行建议

当进入实施轮次时，建议保持同样的团队组织方式：

### 7.1 主线程负责

- 主分支方案协调
- 数据模型冻结
- 关键接口和 ADR 把关

### 7.2 后端小队

- `gateway + metering`
- `marketplace + order`
- `settlement + dispute`

### 7.3 前端小队

- 买方门户
- 卖方控制台
- 平台管理后台

### 7.4 QA / 验证小队

- 接口回归
- 账务对账
- 故障注入
- 性能压测

## 8. 本文档的作用

这份文档不是形式化记录，而是给后续团队解释：

- 为什么这次不是单一仓库二开
- 为什么最终选 `new-api-main` 做主底座
- 为什么同时要借鉴 `coai-main` 和 `litellm-main`
- 为什么项目必须先有规划文件，再进入正式编码
