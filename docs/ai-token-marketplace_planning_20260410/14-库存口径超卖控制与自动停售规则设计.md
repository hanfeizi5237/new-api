# 14-库存口径、超卖控制与自动停售规则设计

更新时间：2026-04-12

## 1. 文档目标

本文件冻结 `M1` 中库存相关的统一口径，包括：

1. `supply_account` 与 `inventory_snapshot` 各字段分别代表什么
2. 可售库存如何计算
3. 下单冻结、支付成功、请求消耗、退款回滚如何影响库存
4. 并发下单时如何避免超卖
5. 商品在什么条件下自动停售、何时允许恢复

## 2. M1 适用边界

- 仅支持 `text/chat` 模型
- 库存单位统一为 `token`
- 一期库存口径以 `supply_account` 为最小供给单元
- 一期不做多 SKU 共享复杂库存池
- 一期不做跨 seller 调拨

## 3. 冻结决策

### 决策 A：库存主口径以 `supply_account` 为中心

- 每个 `supply_account` 对应一条 `inventory_snapshot`
- `listing` 和 `listing_sku` 只是销售视图
- 真正的库存冻结与消耗都落在 `supply_account_id`

### 决策 B：库存分 4 层理解

1. 供给总量
2. 平台认可的可售上限
3. 未支付订单冻结量
4. 已售未消耗承诺量

### 决策 C：`M1` 先保证“不超卖”，再追求“高利用率”

- 可售库存计算要保守
- 风险折扣默认生效
- 验活异常时优先停售

## 4. 字段口径冻结

## 4.1 `supply_account`

| 字段 | 口径定义 |
| --- | --- |
| `TotalCapacity` | 卖家可证明的总供给量或平台录入的总可支配量 |
| `SellableCapacity` | 平台审核后允许对外销售的最大总量 |
| `ReservedCapacity` | 已支付发权但尚未被实际消耗的承诺量 |
| `UsedCapacity` | 已被真实请求消耗的累计量 |
| `VerifyStatus` | 供给可用性验证结果，来自 `seller_secret` 与 channel 探测汇总 |

说明：

- `TotalCapacity` 可大于 `SellableCapacity`
- `ReservedCapacity` 与 `UsedCapacity` 都会永久占用供给
- `ReservedCapacity` 代表平台对买家的未履约余额义务

## 4.2 `inventory_snapshot`

| 字段 | 口径定义 |
| --- | --- |
| `AvailableAmount` | 当前允许新增销售的数量 |
| `FrozenAmount` | 待支付订单已冻结但尚未支付成功的数量 |
| `SoldAmount` | 已支付并已发权的累计销售量 |
| `ConsumedAmount` | 已被真实请求使用的累计量 |
| `RiskDiscountBps` | 风险折扣系数，默认 `10000`，即 100% |
| `HealthScore` | 供给健康度，0-100 |
| `SyncStatus` | `ok/warning/error` |

说明：

- `SoldAmount` 用于报表与稽核，不直接参与余额计算
- `AvailableAmount` 是计算结果，不是自由录入值

## 5. 统一计算公式

## 5.1 原始剩余量

```text
RawRemaining = max(SellableCapacity - ReservedCapacity - UsedCapacity - FrozenAmount, 0)
```

## 5.2 风险折扣后可售量

```text
AvailableAmount = floor(RawRemaining * RiskDiscountBps / 10000)
```

默认值：

- `RiskDiscountBps = 10000`

示例：

- `SellableCapacity = 1,000,000`
- `ReservedCapacity = 100,000`
- `UsedCapacity = 200,000`
- `FrozenAmount = 50,000`
- `RiskDiscountBps = 9000`

则：

- `RawRemaining = 650,000`
- `AvailableAmount = 585,000`

## 5.3 报表口径

```text
SoldAmount = sum(granted_amount)
ConsumedAmount = sum(actual_usage)
OutstandingObligation = ReservedCapacity
```

## 6. 生命周期中的库存变化

## 6.1 创建订单

动作：

- `inventory_snapshot.frozen_amount += freeze_amount`
- 重新计算 `available_amount`

其中：

- `freeze_amount = listing_sku.package_amount * quantity`

## 6.2 支付成功并发权

动作：

- `inventory_snapshot.frozen_amount -= grant_amount`
- `inventory_snapshot.sold_amount += grant_amount`
- `supply_account.reserved_capacity += grant_amount`
- 重新计算 `available_amount`

## 6.3 请求实际消耗

动作：

- `supply_account.reserved_capacity -= actual_quota`
- `supply_account.used_capacity += actual_quota`
- `inventory_snapshot.consumed_amount += actual_quota`
- 重新计算 `available_amount`

说明：

- 实际消耗代表平台已完成履约
- 一旦进入 `used_capacity`，不可再回到 `reserved_capacity`

## 6.4 订单关闭或支付失败

动作：

- `inventory_snapshot.frozen_amount -= freeze_amount`
- 重新计算 `available_amount`

## 6.5 entitlement 退款

`M1` 仅支持未消耗部分退款，动作：

- `supply_account.reserved_capacity -= refundable_amount`
- `inventory_snapshot.sold_amount -= refundable_amount`
- 重新计算 `available_amount`

## 7. 超卖控制规则

## 7.1 下单前校验

下单时必须判断：

- `inventory_snapshot.available_amount >= freeze_amount`

否则直接拒绝下单。

## 7.2 冻结必须原子化

建议使用事务 + 条件更新：

```text
WHERE supply_account_id = ?
AND available_amount >= freeze_amount
```

事务内完成：

1. 读取当前 snapshot
2. 增加 `FrozenAmount`
3. 重新计算 `AvailableAmount`
4. 创建订单

## 7.3 并发胜出规则

- 以先成功提交冻结事务的请求为准
- 后来的请求如条件不满足，返回库存不足
- 不允许“先建单后补冻结”

## 7.4 支付成功后发现库存异常

如果支付成功时发现冻结记录丢失或库存口径异常：

- 本订单仍必须发权
- 平台承担库存差异
- 立即将相关 `listing` 置为 `paused`
- 生成库存异常事件供后台排查

## 8. 自动停售规则

## 8.1 触发 `sold_out`

满足任一条件：

- `available_amount <= 0`
- 所有可售 SKU 均无法满足最小购买量

## 8.2 触发 `paused`

满足任一条件：

- `seller_secret.status != active`
- `supply_account.status != active`
- `VerifyStatus != success`
- `SyncStatus = error`
- `HealthScore < 60`
- 平台人工风控暂停

## 8.3 触发优先级

优先级固定为：

1. `paused`
2. `sold_out`
3. `active`

说明：

- 即便 `available_amount > 0`，只要密钥异常，也必须 `paused`
- `paused` 的优先级高于 `sold_out`

## 9. 恢复上架规则

恢复到 `active` 必须同时满足：

- `seller_secret.status = active`
- `supply_account.status = active`
- `VerifyStatus = success`
- `SyncStatus != error`
- `HealthScore >= 60`
- `available_amount > 0`
- 人工确认恢复

`M1` 冻结：

- 不做自动恢复上架
- 只支持人工恢复

## 10. 定时同步任务建议

建议文件：

- `service/inventory_sync_task.go`

最小任务输入：

- `seller_secret` 最近验活结果
- `Channel.Balance`
- 最近失败率
- `supply_account` 当前容量字段

最小任务输出：

- 更新 `inventory_snapshot`
- 推导 `listing.status`
- 写系统日志

建议频率：

- 5 分钟一次全量巡检
- 支付成功、密钥轮换、验活失败后触发单供给增量刷新

## 11. 风险折扣建议

`M1` 默认值建议：

- `RiskDiscountBps = 9000` 到 `10000`

推荐规则：

- 新卖家默认 `9000`
- 连续稳定后可调到 `9500-10000`
- 异常波动时人工下调

## 12. 与订单、发权、relay 的联动

必须与以下文档保持一致：

- `12-支付订单与权益发放时序设计.md`
- `13-entitlement路由分配与扣减算法设计.md`
- `15-Token与Entitlement绑定及资金源优先级设计.md`

联动规则：

- 订单创建冻结 `FrozenAmount`
- 支付成功发权转 `ReservedCapacity`
- relay 成功结算转 `UsedCapacity`
- 任何环节失败都必须能回写对应库存层

## 13. 不建议的实现

- 用 `Channel.Balance` 直接当可售库存
- 只看卖家声明总量，不做风控折扣
- 先建订单再尝试冻结库存
- 密钥失效后仍保持商品 `active`
- 库存恢复后自动上架，不留人工确认

## 14. 验收标准

满足以下条件时，本专题视为完成：

1. `AvailableAmount` 公式冻结且全链路统一
2. 下单冻结、支付成功、请求消耗、退款回滚四类动作都有明确库存变更
3. 并发下单不会超卖
4. 密钥异常、库存耗尽都能触发自动停售
5. 恢复上架条件明确且 `M1` 只允许人工恢复
6. 文档足以直接指导 `Task 7` 编码
