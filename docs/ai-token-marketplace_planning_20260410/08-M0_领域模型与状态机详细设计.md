# 08-M0 领域模型与状态机详细设计

更新时间：2026-04-12

## 1. 文档目标

本文件用于冻结 `AI Token 集市平台` 在 `new-api-main` 底座上的 M0 详细设计，重点解决以下问题：

1. 一期到底要新增哪些核心实体
2. 每个实体的核心字段、索引、状态机如何定义
3. 账务、幂等、库存、路由责任边界如何落地
4. 在 `new-api-main` 中哪些对象继续复用，哪些对象必须新建

本文件是后续 M1 开发任务书的输入，不是最终代码实现。

## 2. 设计边界

### 2.1 一期明确支持

- 白名单卖家入驻
- 平台托管卖家 Key
- 固定价商品
- `text/chat` 类模型按 `token` 计量的权益包
- 商品审核
- 买方下单支付
- 买方权益发放
- 通过平台代理调用已购权益
- 基础库存同步
- 保障期冻结结算
- 人工争议与人工仲裁

### 2.2 一期明确不支持

- 自由竞价
- 保险服务
- 自动仲裁
- 卖家自助提现
- 图像/音频/task 类型 entitlement 套餐
- 多币种
- 组合包
- 自动调价
- 完全开放卖家市场

## 3. 强制设计约束

### 3.1 平台责任边界

- 平台卖的是“平台代理使用权”，不是卖家原始 API Key
- 买方只能拿平台 `Token`
- 卖家原始 Key 只用于平台运行时路由，不对买方暴露
- 卖家原始凭证主事实源必须是 `seller_secret`，`channels.key` 在 `M1` 仅允许作为运行时镜像
- 争议证据只认平台链路中的 `request_id + Log + usage_ledger`

### 3.2 数据与账务约束

- 金额统一使用整型最小货币单位，例如 `amount_minor`
- token/额度统一使用整型，不用浮点数表示可消费数量
- `M1` 只支持 `text/chat` 模型，额度单位统一冻结为 `token`
- 所有 JSON 存储统一使用 `TEXT`
- 所有 JSON 序列化统一通过 `common.Marshal / common.Unmarshal`
- 所有表设计必须兼容 `SQLite / MySQL / PostgreSQL`
- 不依赖 `JSONB`、`ALTER COLUMN`、数据库方言函数
- 单次请求只能命中一种最终资金来源，不允许在一个请求里混合 entitlement 与 wallet/subscription 扣费

### 3.3 幂等与审计约束

- 支付回调必须有幂等键
- entitlement 创建必须有幂等键
- usage 入账必须有幂等键
- settlement 入账必须有幂等键
- 明文密钥读取与运行时解密必须有审计记录
- 历史事实流水不允许直接改写，只允许补偿修正

## 5. 核心实体详细设计

说明：

- 主键统一使用 `Id int`
- 时间统一使用 `int64` Unix 秒级时间戳，风格对齐现有项目
- 布尔字段优先使用 `bool`
- 扩展字段统一使用 `Extra string` 或 `Meta string` 保存 JSON 文本

## 6. 状态机冻结

### 6.1 `listing.status`

```text
draft -> pending_review -> approved(active) -> paused -> sold_out -> archived
                       \-> rejected
```

规则：

- `approved` 后才允许进入 `active`
- `sold_out` 可由库存同步任务自动触发
- `paused` 可由运营手动触发，也可由供给异常自动触发

### 6.2 `market_order`

拆成三个状态轴：

- `order_status`
  - `pending_payment / paid / closed / disputed / refunded`
- `payment_status`
  - `unpaid / paid / failed / refunded`
- `entitlement_status`
  - `pending / created / partial / failed`

规则：

- `payment_status=paid` 才允许创建 entitlement
- `order_status=closed` 且 `payment_status=unpaid` 表示支付超时关闭
- `order_status=disputed` 不代表 entitlement 不可继续消费，是否冻结由具体规则决定

### 6.3 `entitlement_lot.status`

```text
active -> exhausted
active -> expired
active -> frozen
active -> refunded
```

规则：

- 扣减只从 `active` 批次中发生
- 冻结主要用于争议、退款处理中间态

### 6.4 `settlement_entry.status`

```text
pending -> frozen -> releasable -> settled
                     \-> withheld
pending/frozen/releasable -> reversed
```

规则：

- 支付成功后不立即给卖家结算
- 先生成 `pending/frozen`
- 到达保障期且无争议后进入 `releasable`
- 财务处理后变为 `settled`

### 6.5 `dispute_case.status`

```text
open -> seller_responding -> under_review -> decided -> executed -> closed
```

规则：

- 立案后立即冻结关联 `settlement_entry`
- `executed` 表示退款/扣罚/解冻动作已完成

## 7. 幂等键设计

### 7.1 幂等对象

| 对象 | 幂等键建议 |
| --- | --- |
| 订单创建 | `buyer_id + client_nonce` |
| 支付回调 | `payment_provider + payment_trade_no + callback_stage` |
| entitlement 创建 | `order_no + order_item_id + grant_stage` |
| usage 入账 | `request_id + success_final` |
| settlement 入账 | `usage_event_key + settlement_stage` |
| dispute 立案 | `order_id + dispute_type + opened_at_bucket` |

### 7.2 设计原则

- 幂等键进入业务表，不单独依赖缓存实现
- Redis 锁只能做并发优化，不能替代表级幂等
- 幂等失败要能安全返回“已处理结果”
