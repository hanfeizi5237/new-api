# 19-异步事件、任务重试与补偿机制设计

更新时间：2026-04-16

## 1. 文档目标

本文件用于冻结 Marketplace 在 `new-api-main` 上的异步处理边界、任务重试模型和补偿口径，重点解决：

1. `M1` 首期到底使用什么异步机制
2. 哪些动作必须同步提交，哪些动作允许最终一致性
3. 重试次数、退避策略、死信处理如何统一
4. 异常情况下如何补偿，才能不改写历史事实账
5. 后续支付、库存、发权、usage、结算任务如何复用同一口径

## 2. M1 适用边界

- `M1` 不引入 Kafka、Redis Streams、RabbitMQ 这类新的硬依赖
- `M1` 采用 `数据库事实记录 + 周期扫描任务 + 幂等重放` 的最小实现
- Redis 只允许作为锁和并发优化，不能作为异步事实源
- 支付回调、发权、usage 入账、库存/结算事实落库仍以本地数据库事务为准
- 通知、补扫、过期释放、人工重放允许最终一致性

## 3. 冻结决策

### 决策 A：首期统一采用“DB 驱动任务”而不是消息队列

冻结口径：

- 首期异步统一通过数据库中的任务记录或可扫描业务事实驱动
- 任务执行器风格与现有 `task`、`subscription_reset_task` 一致
- 不要求先建设通用事件总线，先保证任务幂等、可重跑、可追责

### 决策 B：核心资金与权益事实必须同步提交

以下动作不得仅依赖异步：

- 创建订单时冻结库存
- 支付成功回调时更新订单支付状态
- 支付成功回调时创建 `buyer_entitlement / entitlement_lot`
- relay 成功后写 `usage_ledger`
- relay 成功后回写 `entitlement_lot.used_amount`
- entitlement 到期释放时回写 `ReservedCapacity`
- 产生 `settlement_entry` 的事实账写入

### 决策 C：异步只承接“副作用、扫描、修复、重放”

允许异步的动作：

- 订单超时关闭扫描
- entitlement 到期释放扫描
- 库存同步与自动停售
- 卖家 secret 周期验证
- 通知、站内信、统计刷新
- usage finalize 修复
- 人工发权重放、人工结算冲销执行

### 决策 D：补偿只允许“追加修正”或“幂等重放”

禁止动作：

- 直接改写历史 `usage_ledger`
- 直接覆盖旧 `settlement_entry` 的金额
- 直接把已支付订单物理删除
- 直接修改旧 lot 的历史发放事实

允许动作：

- 追加 reversal / compensation 事实
- 针对同一 `event_key` 幂等重放
- 针对未完成动作补跑任务
- 针对未履约部分做释放或退款

### 决策 E：是否重试由错误码家族决定

- 可重试错误：超时、临时网络失败、DB 锁冲突、上游 5xx、任务执行器抢锁冲突
- 不可重试错误：参数错误、权限不足、资源不存在、状态冲突、业务前置条件不满足
- 具体错误码与可重试语义，统一对齐 `20-API契约与错误码规范.md`

## 4. 异步模型建议

## 4.1 推荐统一任务实体

推荐新增 `market_async_task`，用于承接需要显式重试和死信管理的异步动作。

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `Id` | `int` | 主键 |
| `TaskType` | `string` | `close_expired_order/expire_entitlement_lot/inventory_sync/...` |
| `EventKey` | `string` | 幂等键，唯一索引 |
| `AggregateType` | `string` | `order/order_item/entitlement_lot/seller_secret/...` |
| `AggregateId` | `int` | 关联主实体 ID |
| `TriggerSource` | `string` | `system_scan/webhook/manual/admin` |
| `Payload` | `string` | JSON 文本，存最小执行上下文 |
| `Status` | `string` | `pending/running/retrying/succeeded/dead/cancelled` |
| `RetryCount` | `int` | 当前重试次数 |
| `MaxRetryCount` | `int` | 最大重试次数 |
| `NextRunAt` | `int64` | 下次执行时间 |
| `LastErrorCode` | `string` | 最后一次错误码 |
| `LastErrorMessage` | `string` | 最后一次错误摘要 |
| `LockToken` | `string` | 执行锁标识 |
| `LockedAt` | `int64` | 抢占时间 |
| `FinishedAt` | `int64` | 完成时间 |
| `DeadLetterAt` | `int64` | 死信时间 |
| `CreatedAt` | `int64` | 创建时间 |
| `UpdatedAt` | `int64` | 更新时间 |

索引建议：

- 唯一索引：`event_key`
- 普通索引：`status + next_run_at`
- 普通索引：`task_type + status`
- 普通索引：`aggregate_type + aggregate_id`

## 4.2 M1 最小可接受落地

如果 `M1` 暂不引入通用 `market_async_task` 表，最低要求也必须满足：

1. 每类异步任务都有显式幂等键
2. 每类任务都有可扫描的待执行条件
3. 失败后可记录 `retry_count / next_run_at / last_error_code`
4. 死信后可被人工发现和重放
5. 任务补跑不会改写历史事实

## 5. 任务清单冻结

## 5.1 `M1 must-have`

| 任务 | 触发方式 | 目的 |
| --- | --- | --- |
| `close_expired_order` | 定时扫描 | 关闭超时未支付订单并释放 `FrozenAmount` |
| `inventory_sync` | 定时扫描 / 手动触发 | 同步 `inventory_snapshot` 并自动停售 |
| `expire_entitlement_lot` | 定时扫描 | lot 到期时释放 `ReservedCapacity` 并写 expiry settlement |
| `pause_listing_on_secret_disable` | 事件触发 / 扫描兜底 | secret 失效后自动暂停商品 |
| `manual_replay_entitlement` | 人工触发 | 发权失败或审计复核后的补发权 |
| `usage_finalize_repair` | 补扫任务 | 上游已成功但本地 finalize 暂失败时的修复 |

## 5.2 `Pilot MVP` 再补

| 任务 | 触发方式 | 目的 |
| --- | --- | --- |
| `payment_reconcile` | 定时扫描 | 补扫支付平台漏回调、乱序回调 |
| `settlement_release_scan` | 定时扫描 | 冻结期结束后进入 `releasable` |
| `dispute_timeout_scan` | 定时扫描 | 争议超时自动推进到人工队列 |
| `withdrawal_review_timeout` | 定时扫描 | 提现审核超时处理 |

## 6. 同步 / 异步边界

## 6.1 必须同步提交的动作

| 动作 | 原因 |
| --- | --- |
| 下单冻结库存 | 否则会超卖 |
| 支付回调更新订单状态 | 否则支付状态会漂移 |
| 发权创建 lot | 否则买方已付款但无法消费 |
| relay 成功写 usage | 否则无法结算与审计 |
| 扣减/回滚 entitlement 冻结量 | 否则余额失真 |
| 到期释放时写 inventory/settlement 事实 | 否则 `ReservedCapacity` 泄漏 |

## 6.2 允许异步最终一致的动作

| 动作 | 说明 |
| --- | --- |
| 通知、站内信、邮件 | 失败可重试，不影响主账 |
| 页面统计刷新 | 可延迟 |
| 风控报表聚合 | 可按分钟级刷新 |
| secret 周期验证 | 不要求和请求事务绑定 |
| 对账补扫 | 本质是修复，不是主交易事实 |
| 人工补偿执行记录同步 | 允许后台任务完成 |

## 7. 重试策略

## 7.1 默认退避规则

统一建议：

1. 第 1 次失败后：`1 minute`
2. 第 2 次失败后：`5 minutes`
3. 第 3 次失败后：`15 minutes`
4. 第 4 次失败后：`60 minutes`
5. 第 5 次失败后：`6 hours`
6. 超出后进入 `dead`

## 7.2 不可重试条件

以下情况必须直接终止并标记人工处理：

- `MARKET_VALIDATION_*`
- `MARKET_AUTH_*`
- `MARKET_PERMISSION_*`
- `MARKET_RESOURCE_NOT_FOUND`
- `MARKET_STATE_CONFLICT_*`
- `MARKET_ORDER_ALREADY_PAID`
- `MARKET_ORDER_ALREADY_CLOSED`
- `MARKET_LISTING_NOT_ACTIVE`

## 8. 死信与人工处理

进入 `dead` 后必须同时触发：

1. 写 `LastErrorCode / LastErrorMessage`
2. 生成运营可见的告警或待办
3. 如果涉及钱、货、账，生成 `risk_event`
4. 保留人工 `retry` 与 `cancel` 两种动作

人工处理动作要求：

- 受 `17-RBAC、敏感操作与审计权限矩阵设计.md` 约束
- 必须填写 `reason`
- 必须记录 `request_id` 与 `operator`

## 9. 补偿口径

## 9.1 订单超时关闭

补偿动作：

- `market_order.status -> closed`
- `inventory_snapshot.frozen_amount -= freeze_amount`
- 不发权
- 不产生正向 settlement

## 9.2 支付回调重复或乱序

处理原则：

- 同一 `payment_trade_no + callback_stage` 只允许成功一次
- 重复回调返回成功 ACK，但不重复落账
- 若出现“本地已关闭但支付平台报告成功”，不直接自动 reopen
- 应进入 `payment_reconcile` 或人工核查队列

## 9.3 entitlement 到期释放

补偿动作：

- 释放 `ReservedCapacity`
- lot 进入 `expired`
- 生成 `source_type = expiry_release` 的 `settlement_entry`

口径对齐：

- `14-库存口径超卖控制与自动停售规则设计.md`
- `16-结算、佣金、退款与冲销公式设计.md`

## 9.4 relay 成功但本地 finalize 暂失败

处理原则：

- 不允许再次向上游重放真实用户请求
- 应基于 `request_id + Log + usage_ledger event_key` 做本地修复
- 补偿任务只负责“补齐事实账”，不负责“再调用一次上游”

## 9.5 手动退款或赔付

处理原则：

- 未履约部分：释放 `ReservedCapacity`，不新增正向结算
- 已确认收入部分：追加 reversal entry
- 不直接改旧 settlement 记录

## 10. 在 new-api-main 中的落点建议

推荐文件：

- `model/market_async_task.go`
- `service/async_task_service.go`
- `service/close_expired_order_task.go`
- `service/expire_entitlement_task.go`
- `service/inventory_sync_task.go`
- `service/usage_finalize_repair_task.go`

如果 `M1` 暂不建统一任务表，也至少应在：

- `service/inventory_sync_task.go`
- `service/market_payment_service.go`
- `service/entitlement_service.go`
- `service/usage_ledger_service.go`

中保留统一的重试、死信、审计语义。

## 11. 验收标准

满足以下条件时，本专题视为完成：

1. 已明确 `M1` 首期不引入 MQ，采用 DB 驱动任务模型
2. 已明确哪些动作必须同步、哪些允许异步
3. 已冻结默认重试、死信和人工重放规则
4. 已明确补偿只允许追加修正，不允许改写历史事实
5. 文档足以指导后续 `close_expired_order / inventory_sync / expire_entitlement_lot / usage_finalize_repair` 的实现
