## P0-1：修复 `lockForUpdate`

**文件**

- `service/marketplace_inventory.go`

**当前问题**

当前实现会在 SQLite / MySQL / PostgreSQL 三种数据库上全部直接 `return query`，等于所有受支持数据库都没真正执行 `FOR UPDATE`。

**影响范围**

- `service/market_order_service.go`
- `service/market_payment_service.go`
- `service/entitlement_service.go`
- `service/marketplace_entitlement_funding.go`
- `service/inventory_sync_task.go`

**风险**

- 下单冻结库存并发不安全
- webhook 并发下订单状态竞争
- entitlement 预冻结 / 结算 / 退款可能串扰

**修复要求**

- SQLite：跳过 `FOR UPDATE`
- MySQL / PostgreSQL：保留 `FOR UPDATE`

**建议写法**

```go
func lockForUpdate(query *gorm.DB) *gorm.DB {
    if common.UsingSQLite {
        return query
    }
    return query.Set("gorm:query_option", "FOR UPDATE")
}
```

**验收标准**

- SQLite 不报 `FOR UPDATE` 语法错误
- MySQL / PostgreSQL 查询带 `FOR UPDATE`
- marketplace 相关事务重新具备基础串行保护

## P0-2：修复“支付成功但发权失败”状态落库问题

**文件**

- `service/market_payment_service.go`

**当前问题**

当前逻辑在同一个事务里：

1. 把订单改成 `paid`
2. 发权失败后把 `entitlement_status` 改成 `failed`
3. 直接 `return err`

由于事务最终返回 error，这些状态更新会一起回滚。

**设计要求**

根据 `12-支付订单与权益发放时序设计.md`，支付成功但发权失败时，数据库应保留：

- `payment_status=paid`
- `order_status=paid`
- `entitlement_status=failed`

**风险**

- PSP 已确认成功，但平台库里仍像没支付
- 后续无法按真实失败态做重放
- 客服与补偿处理口径失真

**修复要求**

- 支付成功状态必须能落库保留
- 发权失败时必须能稳定保留 `entitlement_status=failed`
- 后续重试只补做发权，不能再次累加库存

**验收标准**

- 能稳定构造 `paid + entitlement_failed`
- 后续重试只重放 entitlement grant
- 不重复增加 `InventorySnapshot.SoldAmount`
- 不重复增加 `SupplyAccount.ReservedCapacity`

## P0-3：支付 Webhook 端点无 body size 限制

**文件**

- `controller/market_payment.go` (lines 89, 157, 216)
- `router/api-router.go` (lines 205-209)

**当前问题**

所有 4 个支付 webhook handler (epay / stripe / creem / waffo) 直接调用 `io.ReadAll(c.Request.Body)` 无上限，且 `marketRoute` 未挂载 body size 限制中间件。

**风险**

- 攻击者可发送超大 POST body 导致服务器 OOM
- 这是公开端点，无认证保护

**修复建议**

在 handler 内包装 request body：

```go
c.Request.Body = http.MaxBytesReader(c.Writer, c.Request.Body, 1<<20) // 1MB
```

或为 webhook 路由添加 `middleware.RequestBodyLimit(1<<20)` 中间件。

**验收标准**

- POST body 超过 1MB 时返回 413
- 正常支付回调 body 不受影响（通常 < 50KB）

## P0-4：`PayMarketOrder` 未校验 `payment_method`

**文件**

- `controller/market_order.go:173-176`

**当前问题**

```go
type PayMarketOrderRequest struct {
    PaymentMethod string `json:"payment_method"`
}
```

任意字符串直接传入 `service.PrepareMarketOrderPayment`，缺乏快速失败。

**风险**

- 非法 payment_method 穿透到 service 层，可能返回不清晰的错误
- 缺乏 API 边界防护

**修复建议**

Controller 层白名单校验：

```go
if req.PaymentMethod != "epay" && req.PaymentMethod != "stripe" &&
   req.PaymentMethod != "creem" && req.PaymentMethod != "waffo" {
    common.ApiError(c, errors.New("invalid payment_method"))
    return
}
```

**验收标准**

- 传入非法 payment_method 时返回 400
- 传入合法值时正常进入 service 层

## P1-1：补写 `market_order_item.used_amount`

**文件**

- `service/marketplace_entitlement_funding.go`
- `model/market_order.go`

**当前问题**

目前 success settle 只更新了：

- `entitlement_lot.used_amount`
- `buyer_entitlement.total_used`
- `supply.used_capacity`
- `inventory_snapshot.consumed_amount`

但没有更新：

- `market_order_item.used_amount`

**为什么是问题**

M1 任务书要求结算成功后同步回写订单项消耗量。否则订单项维度无法作为结算、争议、后台对账的直接依据。

**修复要求**

- success settle 时，累加对应 `order_item_id` 的 `used_amount`
- failed / refund 不得误写
- retry 不得重复累加

**验收标准**

- 单次成功请求后，`market_order_item.used_amount` 正确增加
- retry 成功不会重复累计
- failed / refund 不产生脏数据

## P1-2：补全 `usage_ledger` 的 usage 明细

**文件**

- `model/usage_ledger.go`
- `service/marketplace_entitlement_funding.go`

**当前问题**

`UsageLedger` 模型已有：

- `prompt_tokens`
- `completion_tokens`
- `total_tokens`
- `other`

但当前 marketplace 写账只写了归属和 quota，没有把真实 usage 明细写进去。

**风险**

- 只能证明“扣了多少配额”
- 不能证明“真实消耗了多少 prompt/completion/total tokens”
- 审计、争议、结算证据不足

**修复要求**

- success ledger 写入 usage 明细
- failed ledger 至少写：
  - `ledger_status`
  - `error_code`
  - request 归属字段
- 如有必要，同步 marketplace 归属信息到 `other`

**验收标准**

- success ledger 具备 token usage 明细
- failed ledger 具备失败归属信息
- `usage_ledger` 可独立支撑最小审计追溯

## P1-5：`CompleteMarketOrderPayment` 在事务外重复查询 items

## P2：优化 N+1 查询

**文件**

- `controller/listing.go`
- `controller/market_order.go`

**当前问题**

存在逐条查详情的典型 N+1：

- public listings 逐条查 detail
- admin listings 逐条查 sku
- buyer orders 逐条查 items

**影响**

- 数据量小时不明显
- 数据量上来后 marketplace 页和后台页会变慢

**修复要求**

- 改为批量查询后组装返回
- 避免 controller 层按记录循环打 DB

**验收标准**

- 单页 20 条数据时查询次数明显下降
- controller 层不再出现逐条 detail/item 查询
