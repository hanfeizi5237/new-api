### Issue #3 — SQLite 不兼容 `FOR UPDATE`

**状态**: ❌ **未修复**

仍然出现在以下所有文件：

| 文件 | 出现次数 |
|------|----------|
| `service/entitlement_service.go` | 1 处（line 45） |
| `service/market_order_service.go` | 4 处（lines 165, 169, 257, 323） |
| `service/market_payment_service.go` | 3 处（lines 131, 163, 167, 243） |
| `service/marketplace_entitlement_funding.go` | 3 处（lines 67, 123, 127） |
| `service/inventory_sync_task.go` | 2 处（lines 95, 156） |

**修复方案**（已在原报告中给出）: 在每个 `FOR UPDATE` 查询前加 SQLite 分支：
```go
query := tx.Where(...)
if !common.UsingSQLite {
    query = query.Set("gorm:query_option", "FOR UPDATE")
}
err := query.First(&record).Error
```

注意：`common.UsingMySQL` 和 `common.UsingPostgreSQL` 也需要分支判断。更简洁的写法：
```go
if common.UsingSQLite {
    err = tx.Where(...).First(&record).Error
} else {
    err = tx.Set("gorm:query_option", "FOR UPDATE").Where(...).First(&record).Error
}
```

### Issue #4 — 超卖竞态（非原子读-改-写）

**状态**: ❌ **未修复**

**文件**: `service/market_order_service.go:163-194`

仍采用：读 snapshot → 内存检查 `AvailableAmount >= freezeAmount` → 修改内存 → `Save` 的模式。两个并发事务可在 FOR UPDATE 行锁生效前读到相同的 `AvailableAmount`，导致超卖。

**根因**: 即使 `FOR UPDATE` 在 PostgreSQL/MySQL 下能正确加锁，但检查条件 `snapshot.AvailableAmount` 是内存变量而非 SQL WHERE 条件。事务 B 等待事务 A 的锁释放后，会读到事务 A 修改后的值——所以实际上在 PostgreSQL/MySQL 下 `FOR UPDATE` 能保证串行化，这个竞态只在 SQLite 下因没有行锁而真正发生。

**建议**: 先修复 Issue #3（SQLite FOR UPDATE），然后改用原子 SQL 作为双重保障：
```sql
UPDATE inventory_snapshots
SET frozen_amount = frozen_amount + ?,
    available_amount = (sellable_capacity - reserved_capacity - used_capacity - (frozen_amount + ?)) * risk_discount_bps / 10000
WHERE supply_account_id = ?
  AND available_amount >= ?
```

## 二、新增代码审查

### marketplace_entitlement_funding.go — 潜在问题

#### N1: `listCandidateMarketplaceLotsTx` 存在 N+1 查询（严重）

**文件**: `service/marketplace_entitlement_funding.go:394-456`

`listCandidateMarketplaceLotsTx` 先查询所有 `EntitlementLot`（无模型名过滤），然后对每个 lot 分别查询：
- `getBuyerEntitlementByIDTx`（检查 model_name）
- `model.GetListingByID`（检查 status）
- `model.GetSupplyAccountByID`（检查 status）
- `tx.Model(&model.SellerSecret{}).Count(...)`（检查 active secret）

一个用户有 10 个 lot 时，会产生 1 + 10 × 4 = 41 次数据库查询。

**建议**: 用 JOIN 或批量查询替代。在 `listCandidateMarketplaceLotsTx` 中直接用 JOIN 过滤模型名、listing 状态、supply 状态，减少到 1-2 次查询。

#### N2: `resolveMarketplaceLotChannelIDsTx` 循环查询 channel

**文件**: `service/marketplace_entitlement_funding.go:466-489`

对每个 binding 调用 `model.GetChannelById(binding.ChannelId, true)`。如果 supply account 绑定了多个 channel，会产生多次查询。

**建议**: 批量查询所有 channel，然后内存过滤。

#### N3: `MarketplaceEntitlementFunding.Settle` 中 `additionalNeeded` 检查不完整

**文件**: `service/marketplace_entitlement_funding.go:141-145`

```go
if additionalNeeded > 0 {
    remainingAvailable := lot.GrantedAmount - lot.UsedAmount - lot.RefundedAmount - lot.FrozenAmount
    if remainingAvailable < int64(additionalNeeded) {
        return errInsufficientMarketplaceEntitlement
    }
}
```

`lot.FrozenAmount` 在此处已经被行 148 的 `frozenRelease` 减少前读取，但 `lot.FrozenAmount` 的更新在行 148。这里的 `remainingAvailable` 计算用的是**更新前**的 `FrozenAmount`，这是正确的——因为 frozen 即将释放，释放后的额度仍可用于本次结算。但如果 `additionalNeeded` 很大，释放的 frozen 不足以覆盖本次增加，检查仍然正确。

**结论**: 逻辑正确，但可读性差。建议添加注释说明为什么用更新前的 FrozenAmount。

#### N4: `billing_session.go:46` — marketplace 的 `delta == 0` 处理

```go
if delta == 0 && s.funding.Source() != BillingSourceMarketplaceEntitlement {
    s.settled = true
    return nil
}
```

当 marketplace 的 `delta == 0`（预扣费 = 实际消耗），代码不会提前 return，会继续执行 `funding.Settle(0)`。查看 `MarketplaceEntitlementFunding.Settle(0)` 的逻辑：
- `actualQuota = preConsumed + 0 = preConsumed`
- `frozenRelease = min(FrozenAmount, preConsumed)` → 释放全部冻结
- `actualQuota > 0` → 累加 UsedAmount

这意味着即使 `delta == 0`，marketplace 的 Settle 仍然会执行库存更新。这是**有意设计**——确保 UsageLedger 被正确写入。代码正确。
