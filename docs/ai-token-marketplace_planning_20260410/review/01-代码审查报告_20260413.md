## 二、阻断级缺陷（必须修复后才能上线）

### 1. SKU 验证条件永真 — 所有 SKU 被拒绝

**文件**: `service/listing_service.go:170`

```go
if skus[i].PackageUnit != "token" || skus[i].PackageUnit != supply.QuotaUnit {
    return nil, nil, errors.New("M1 only supports token-based sku units")
}
```

**根因**: `supply.QuotaUnit` 已被行 98 强制为 `"token"`。条件等价于 `A != "token" || A != "token"`，永远为 true。

**影响**: `CreateListingWithSKUs` 接口完全不可用，无法创建任何商品。

**修复**: 删除 `|| skus[i].PackageUnit != supply.QuotaUnit`，或直接删除整段检查（行 167-168 已做默认值填充）。

### 2. Stripe 回调金额解析静默失败 — 零元发货漏洞

**文件**: `controller/market_payment.go:134`

```go
amountMinor, _ := strconv.ParseInt(event.GetObjectValue("amount_total"), 10, 64)
```

**根因**: `_` 忽略 parse error。若 Stripe 返回非数字金额（如空字符串），`amountMinor = 0`。

**影响**: `CompleteMarketOrderPayment` 的金额校验 `PayableAmountMinor > 0` 不会触发（0 不大于 0），订单以零元完成支付并发货。

**修复**:
```go
amountMinor, err := strconv.ParseInt(event.GetObjectValue("amount_total"), 10, 64)
if err != nil {
    log.Printf("Marketplace Stripe amount parse error: %v", err)
    c.AbortWithStatus(http.StatusBadRequest)
    return
}
```

### 3. SQLite 不兼容 — `FOR UPDATE` 导致 SQL 报错

**影响文件**:
- `service/entitlement_service.go:45`
- `service/market_order_service.go:165,169,306,310`
- `service/market_payment_service.go:131,158,162,235`

**根因**: SQLite 不支持 `SELECT ... FOR UPDATE`。GORM 的 `gorm:query_option` 会直接追加到 SQL，在 SQLite 下产生语法错误。

**影响**: 使用 SQLite 部署时，下单、支付、权益发放全部失败。

**修复**: 每个 `FOR UPDATE` 查询前加 SQLite 分支：
```go
query := tx.Where(...)
if !common.UsingSQLite {
    query = query.Set("gorm:query_option", "FOR UPDATE")
}
err := query.First(&record).Error
```

### 4. 超卖竞态 — `SupplyAccount` 容量扣减非原子

**文件**: `service/market_order_service.go:163-194`

**根因**: 订单创建时在事务内对 `SupplyAccount` 和 `InventorySnapshot` 执行 `SELECT ... FOR UPDATE` → 读内存值 → 判断 `AvailableAmount >= freezeAmount` → 修改内存值 → `Save`。

问题在于两个并发事务：
1. 事务 A 读取 `snapshot.AvailableAmount = 10000`
2. 事务 B 读取 `snapshot.AvailableAmount = 10000`（A 尚未提交）
3. 事务 A 冻结 6000，通过检查，写入 `FrozenAmount += 6000`
4. 事务 B 冻结 6000，通过检查（仍看到 AvailableAmount = 10000），写入 `FrozenAmount += 6000`
5. 实际冻结 12000 > 实际可用 10000 → 超卖

**影响**: 高并发下单场景下，库存冻结量超过实际可用量。

**修复**: 改用原子 SQL 更新：
```sql
UPDATE inventory_snapshots
SET frozen_amount = frozen_amount + ?
WHERE supply_account_id = ?
  AND (sellable_capacity - reserved_capacity - used_capacity - frozen_amount) * risk_discount_bps / 10000 >= ?
```

### 5. 支付重试导致库存数据损坏

**文件**: `service/market_payment_service.go:136,155-183`

**场景**: 支付成功但权益发放失败（`PaymentStatus = "paid"`, `EntitlementStatus = "failed"`），管理员或系统重试 `CompleteMarketOrderPayment`。

**根因**: 行 136 的早退条件只覆盖 `paid + created`。`paid + failed` 会走到行 155-183，重新执行：
```go
snapshot.SoldAmount += grantAmount     // 重复累加
supply.ReservedCapacity += grantAmount // 重复累加
```

**修复**: 增加 `paid + failed` 的单独处理路径——只重试 `grantEntitlementsForOrderTx`，跳过库存操作（库存已在首次支付时正确修改）。

## 三、严重缺陷

### 6. 支付发起接口缺少限流

**文件**: `router/api-router.go:200`

```go
marketSelfRoute.POST("/orders/:id/pay", controller.PayMarketOrder)
```

对比现有支付路由（全部有 `middleware.CriticalRateLimit()`）：
```
selfRoute.POST("/pay", middleware.CriticalRateLimit(), ...)
selfRoute.POST("/stripe/pay", middleware.CriticalRateLimit(), ...)
selfRoute.POST("/creem/pay", middleware.CriticalRateLimit(), ...)
selfRoute.POST("/waffo/pay", middleware.CriticalRateLimit(), ...)
```

市场支付接口是唯一缺少限流的支付入口。恶意用户可刷请求创建大量第三方 checkout session，消耗 API 配额。

**修复**:
```go
marketSelfRoute.POST("/orders/:id/pay", middleware.CriticalRateLimit(), controller.PayMarketOrder)
```

### 7. 权益发放 `FOR UPDATE` 在 SQLite 下报错（同 #3 的细化）

**文件**: `service/entitlement_service.go:45`

这是最隐蔽的一处——它不在下单/支付主链路，而是在权益发放路径。即使下单时通过行锁保证了安全，这一处仍会导致 SQLite 部署下权益发放完全失败。

## 四、设计缺陷

### 8. N+1 查询 — 列表接口性能退化

| 文件 | 行号 | 问题 |
|------|------|------|
| `controller/listing.go` | 41-48 | `GetMarketListings` 循环调 `GetPublicMarketListingDetail`（每次 2 次 DB 查询） |

10 个 listing = 20+ 次 DB 查询。100 个 = 200+ 次。

### 9. 状态值全部为魔法字符串

全部状态值散落在代码中，无任何常量定义：

| 字段 | 使用的值 | 定义位置 |
|------|----------|----------|
| `Listing.Status` | `paused/active/sold_out/archived` | 多处硬编码 |
| `Listing.AuditStatus` | `draft/pending_review/approved/rejected` | 多处硬编码 |
| `SellerSecret.Status` | `draft/active/disabled/verifying/rotating` | 多处硬编码 |
| `SupplyAccount.Status` | `active/paused` | 多处硬编码 |
| `EntitlementLot.Status` | `active/frozen/exhausted/refunded` | 多处硬编码 |
| `MarketOrderItem.Status` | `pending_payment/granted/closed` | 多处硬编码 |

一旦拼写错误（如 `"actve"`），状态机将无声失效。

**建议**: 创建 `constant/marketplace.go`，定义所有状态常量。

### 10. `inventory_sync_task.go` 不存在 — 自动停售机制缺失

设计文档 `14-库存口径超卖控制与自动停售规则设计.md` §10 和 `09-M1 交易闭环开发任务书.md` Task 7 都定义了定时同步任务：

1. 定时调用上游 API 验证密钥有效性
2. 更新 `inventory_snapshot` 的 `HealthScore` 和 `SyncStatus`
3. 根据规则自动暂停/恢复 listing

**实际状态**: 文件不存在。意味着：
- 密钥失效后 listing 不会自动下架
- 上游额度变化后库存不同步
- `listing.status` 永远只能手动修改

## 六、代码质量与合规

### 正面评价

| 项 | 说明 |
|----|------|
| 密钥加密体系 | AES-256-GCM + 随机 nonce + JSON 信封 + HMAC 指纹，实现正确 |
| 事务管理 | 订单创建、支付完成、密钥操作均在事务内，`FOR UPDATE` 行锁思路正确 |
| 幂等设计 | 订单 `idempotency_key`、权益 `source_event_key`、UsageLedger `event_key` 三层幂等 |
| 金额安全 | 全部 `int64` minor 单位，无浮点金额 |
| 视图脱敏 | `SellerSecretView` 不返回密文和明文，只返回 `MaskedValue` + `Fingerprint` |
| 测试覆盖 | 21 个测试文件，覆盖核心路径 |
| JSON 规范 | 遵循 CLAUDE.md Rule 1，使用 `common.Marshal/UnmarshalJsonStr` |

### 安全与合规风险

| 风险 | 位置 | 说明 |
|------|------|------|
| 密钥明文内存 | `secret_runtime_resolver.go` | 解密后 key 存在于 Go 内存，GC 后回收。生产环境可接受 |
| 指纹盐无轮换 | `SELLER_SECRET_FINGERPRINT_SALT` | 盐泄露后可离线做密钥碰撞检测 |
| Webhook 无 IP 白名单 | `market_payment.go` | 仅依赖签名验证 |
| 合规风险 | 需求规格书 §12 | API Key 二级交易违反多数 AI 厂商 ToS，需做好隔离和免责 |
