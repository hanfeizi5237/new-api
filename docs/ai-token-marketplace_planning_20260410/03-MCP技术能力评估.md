# 03-MCP技术能力评估

## 1. 说明

本次任务的核心输入来自本地代码仓与需求文档，不依赖外部 MCP 服务也能完成底座分析与架构规划。

但从“实施落地”的角度，仍然需要明确：

- 系统需要哪些技术能力
- 候选底座已经覆盖了哪些能力
- 还需要补哪些基础设施和组件

因此本文件把 “MCP 能力评估” 转译为 “技术能力评估 + 基础设施补位清单”。

## 3. 候选底座覆盖评估

## 4. 本项目真正需要补位的基础设施

即使选择 `new-api-main` 为主底座，仍需补以下基础设施：

### 4.1 数据层

- 主库：MySQL 或 PostgreSQL
- 缓存：Redis
- 异步任务：Kafka / RabbitMQ / Redis Streams 三选一
- 对象存储：MinIO 或云存储
- 审计与检索：ClickHouse 或 ELK 可选

推荐一期配置：

- MySQL
- Redis
- Redis Streams 或 Kafka
- MinIO

### 4.2 密钥与加密

必须增加：

- 应用级字段加密
- 独立 Seller Key Vault 抽象
- 密钥使用审计
- 定期轮换与验证任务

推荐实现：

- 一期：数据库密文 + KMS 主密钥 + 服务内解密
- 二期：独立 Vault 服务

### 4.3 账务与流水

现有项目中的额度与日志机制不够支撑市场化账务，需要新增：

- `wallet_ledger`
- `inventory_ledger`
- `usage_ledger`
- `settlement_ledger`
- `refund_ledger`

### 4.4 工作流与异步任务

以下行为不应全部同步完成：

- 卖家密钥轮询和健康检查
- 商品库存刷新
- 用量归档
- 结算出账
- 风控规则命中
- 争议 SLA 提醒

## 5. 目标技术栈建议

## 5.1 应用层

- 后端：Go
- Web 前端：React
- ORM：GORM
- API：REST 为主，内部可保留服务层接口

## 5.2 数据与消息

- MySQL：交易主库
- Redis：缓存、锁、短期队列
- Kafka 或 Redis Streams：异步事件总线

## 5.3 运维与观测

- Prometheus + Grafana
- Loki / ELK
- Sentry
- OpenTelemetry

## 5.4 安全与合规

- KMS / Vault
- 字段级加密
- 访问审计
- 管理员敏感操作二次验证

## 6. 替代方案评估

### 6.1 如果强行选 `coai-main` 做主底座

替代收益：

- 前台体验更快成型

替代代价：

- 后端交易域大面积重构
- 路由与账务基础没有 `new-api-main` 那么整齐

结论：

- 不推荐

### 6.2 如果强行选 `litellm-main` 做主底座

替代收益：

- 企业治理和多模型能力一步到位

替代代价：

- 技术栈切换
- 大量市场业务从零补
- 首期研发风险明显上升

结论：

- 不推荐

## 7. 推荐的技术路线

推荐路线不是“平台只依赖一个仓库”，而是：

- 主实现基于 `new-api-main`
- 交互体验和运营页面参考 `coai-main`
- 路由治理、预算、审计参考 `litellm-main`

这是一条“底座明确、能力借鉴分层”的路线，而不是“多仓拼装上线”的路线。

## 8. 首期不建议引入的技术

为了避免工程失控，首期不建议引入：

- 微服务全拆分
- 多数据库分域强拆
- 复杂规则引擎平台
- 自动仲裁 AI 决策
- 多区域多活

这些都可以在第一批真实交易数据稳定后再推进。

## 9. 最终结论

从技术能力视角看，`new-api-main` 已经覆盖了：

- 网关
- 用户/Token/Group
- 支付/充值/订阅
- 后台/日志/运维

它未覆盖的是：

- 卖家市场
- 商品库存
- 交易结算
- 风控仲裁

而这恰好是本项目应新增的业务层，不是应该用另一个项目取代它的理由。
