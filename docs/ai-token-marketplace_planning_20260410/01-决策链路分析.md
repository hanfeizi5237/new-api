# 01-决策链路分析

## 1. 任务概述

### 1.1 用户原始诉求

用户希望基于现有代码仓，完成一份面向 `AI Token 集市平台` 的全面分析、系统设计和可实施落地方案，同时明确 `coai-main`、`new-api-main`、`litellm-main` 三个项目中是否存在适合作为系统底座的项目。

### 1.2 任务类型识别

本任务同时属于以下类型：

- 架构设计
- 技术选型
- 底座评估
- 项目规划
- 实施路线设计

### 1.3 复杂度评估

复杂度评估为：`高`

原因：

1. 需求不是单系统，而是交易平台、网关、账务、风控的组合
2. 需要横向比较三个仓库的架构成熟度与改造成本
3. 需要产出不仅能“讲清楚”，还要能“拿去开工”

## 2. 需求拆解结果

从需求规格说明书中抽出的关键能力如下：

### 2.1 平台侧硬需求

- 卖家可上架 AI 厂商/模型/额度商品
- 买家可搜索、比价、购买、管理资产
- 平台发放统一 Client Key 和 Endpoint
- 所有调用经平台 Relay
- 平台按实际使用量计费扣减
- 平台负责结算、风控、争议处理和后台运营

### 2.2 非功能硬需求

- 统一 API 兼容官方格式
- SSE 流式完整支持
- 高可用路由与故障切换
- API Key 加密存储
- 最终一致性账务
- 横向扩展与监控告警

### 2.3 隐含需求

隐含但关键的需求有三项：

1. 卖家密钥不能暴露给买家
2. 商品售卖库存与实际用量必须能核对
3. 平台必须能区分“预售权益”和“真实消耗”

这三项决定了该系统不是传统意义上的充值站，也不是普通 OpenAI 代理站。

## 3. 底座评估维度

本次评估采用以下六个维度：

| 维度 | 权重 | 解释 |
| --- | --- | --- |
| 网关与协议兼容成熟度 | 25% | 是否具备统一 Relay、路由、重试、模型兼容 |
| 钱包/支付/计费基础 | 20% | 是否已有用户额度、支付订单、充值/订阅、日志 |
| 后台与门户成熟度 | 15% | 是否已有可复用管理后台和用户控制台 |
| 交易市场适配性 | 20% | 是否接近卖家、商品、订单、结算、争议语义 |
| 技术栈连续性 | 10% | 是否与当前团队和现有仓栈一致 |
| 二开复杂度 | 10% | 从现状改造成目标平台的代价 |

## 5. 评分结论

| 仓库 | 网关成熟度 | 计费基础 | 门户后台 | 交易适配 | 技术栈 | 二开难度 | 综合判断 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| `new-api-main` | 高 | 高 | 中高 | 中低 | 高 | 中高 | 第一选择 |
| `coai-main` | 中高 | 中高 | 高 | 低 | 高 | 中 | 第二选择 |
| `litellm-main` | 很高 | 中 | 中 | 很低 | 低 | 低 | 第三选择 |

注：

- 这里的“二开难度”分值表示“越高越容易二开”
- `new-api-main` 胜在离目标平台基础设施最近
- `coai-main` 胜在用户产品面
- `litellm-main` 胜在企业治理面

## 6. 关键 ADR 决策

### ADR-001：项目一期采用模块化单体，不采用微服务

原因：

- 当前最佳底座 `new-api-main` 本身就是模块化单体
- 平台第一阶段最大风险在于领域不清，而非拆分不够
- 微服务会把“领域错误”放大成“跨服务协调错误”

### ADR-002：主底座选 `new-api-main`

原因：

- 可直接承接统一 Relay、额度、支付、后台
- Go 技术栈连续
- 改造成本最低
- 文档完整度高，便于团队接手

### ADR-003：`coai-main` 不做主底座，只做体验参考

原因：

- 强项是前台体验和运营产品感
- 业务中心不是 C2C 交易市场
- 若强行以其为核心，后端交易域改造会反噬

### ADR-004：`litellm-main` 不做主底座，只抽取设计思想

原因：

- 可参考其团队、预算、Spend、Guardrails 模型
- 不值得在一期为了这些能力切换整套栈

### ADR-005：平台以“买方权益账户”而非“卖方原始 key”对外暴露

原因：

- 这是合规、审计和争议处理的前提
- 也是平台能做担保、清算、风险分级的前提

### ADR-006：账务采用“预扣 + 实扣 + 结算修正”模型

原因：

- 流式、长文本、图像、异步任务天然存在估算误差
- 需要允许先预留、后对账、再结算

## 7. 可实施方案结论

最终建议不是“把需求文档塞进某个项目”，而是：

1. 在 `new-api-main` 上建立实施主线
2. 新增 `seller / catalog / order / entitlement / settlement / dispute` 六个大域
3. 借鉴 `coai-main` 的前台体验和运营页面组织方式
4. 借鉴 `litellm-main` 的预算、审计、fallback、团队治理设计
5. 先做 `MVP-Lite`，不要直接承诺需求文档中的完整自由市场版本

## 8. 潜在风险

在进入实施前必须正视以下现实：

- 上游供应商是否允许额度 resale / delegated access，需要单独法务审查
- 卖家密钥真实性与可售额度真实性不能只靠一次校验
- “固定包售卖”与“真实 token 实时消耗”之间天然存在核算差异
- 如果没有担保期与仲裁，C2C 模式很容易在首批用户中失去信任

## 9. 最终建议

建议把项目目标切成两层：

### 9.1 第一层：验证市场闭环

- 平台托管卖家
- 人工审核上架
- 固定价格商品
- 买方统一 Key 调用
- 真实用量扣减
- 手工仲裁

### 9.2 第二层：扩大平台开放度

- 自助卖家入驻
- 自助上架
- 信用体系
- 自动冻结/结算
- 争议工作流
- 更多支付与提现渠道

这能显著降低首期失败概率。
