## 3. 三层定义

## 3.1 MVP-Lite

目标：

- 验证“商品成交 -> entitlement 发放 -> 平台 relay 消费”闭环

冻结范围：

- 白名单卖家
- 人工审核
- 平台托管密钥
- 固定价商品
- 单商品单 SKU 下单
- entitlement 单资金源消费
- 人工处理异常

## 3.2 Pilot MVP

目标：

- 在真实白名单卖家/买家下稳定试运行

新增能力：

- settlement release
- dispute 工单闭环
- 对账、止损、灰度 runbook
- 更完整的运营后台

## 3.3 Marketplace V1

目标：

- 从平台代运营市场演进到更开放的 marketplace

新增能力：

- 更开放的卖家入驻
- 更完整的 seller self-service
- 信用分、风控自动化
- 更丰富的价格/商品/活动模型

## 5. 当前 M1 验收口径

`M1` 是否完成，只看以下事项：

1. seller / listing / seller_secret 基础管理
2. market order 创建与支付回调幂等
3. entitlement 发放
4. entitlement 接入 relay 消费
5. 单请求不双扣 token / wallet / subscription
6. 库存同步与自动停售
7. 基础后台页面与买方页面可演示

以下事项不应作为 `M1` 失败判定：

- 自由竞价
- 卖家自助提现
- 自动仲裁
- 完整 dispute 闭环
- settlement release
- 信用分
- 复杂促销和组合包

## 6. 与原始需求的差异说明

## 6.1 市场开放度

原始需求偏向“开放市场”；当前实施基线是“平台托管型 marketplace”。

原因：

- 合规风险高
- 卖家供给真实性需要平台兜底
- 首期账务与仲裁能力不足以支撑完全开放

## 6.2 商品模型

原始需求覆盖竞价、订阅、组合包、自动调价；当前仅冻结固定价商品。

原因：

- 先保证库存、支付、发权、扣减四条链正确
- 避免价格模型复杂度反噬账务模型

## 6.3 结算与争议

原始需求把结算、仲裁、信用都写成平台标配；当前只把这些能力拆成后续阶段交付。

原因：

- 这些能力依赖先有稳定的事实账和审计口径

## 7. 变更控制规则

后续如有新增需求，必须显式标注为以下之一：

- `M1 must-have`
- `Pilot MVP`
- `Marketplace V1`
- `Future backlog`

不允许再使用“需求文档里提过”作为直接插入 `M1` 的理由。

## 8. 对评审和验收的要求

在评审或汇报时，必须明确使用以下口径：

- “完整需求愿景”引用原始需求文档
- “当前实现范围”引用本文件和 `09-M1_交易闭环开发任务书.md`
- “阶段缺口”按后续 phase 归档，而不是直接记为实现缺陷

## 9. 验收标准

满足以下条件时，本专题视为完成：

1. 当前阶段与完整愿景之间的边界已被写清
2. `M1` 验收不再和原始完整需求直接冲突
3. 评审人员可用本文件区分“未做”与“后续阶段再做”
4. 文档足以作为后续立项、周报和阶段验收口径
