# 04-完整执行步骤

## 1. 总体实施策略

建议按照 `MVP-Lite -> Pilot MVP -> Marketplace V1` 三段推进，而不是一次性交付完整需求文档中的全部能力。

## 2. Phase 拆解

| Phase | 目标 | 输入 | 输出 | 预计耗时 |
| --- | --- | --- | --- | --- |
| Phase 0 | 冻结边界与主底座 | 需求文档、底座评估 | 数据模型草案、ADR、范围确认 | 1-2 周 |
| Phase 1 | 接管网关与密钥托管基础 | `new-api-main` | Seller Key Vault、Channel 扩展、路由骨架 | 2-3 周 |
| Phase 2 | 建立市场与订单主流程 | Phase 1 输出 | 商品、订单、支付、买家权益闭环 | 3 周 |
| Phase 3 | 打通计费、结算与库存 | Phase 2 输出 | 三本账、结算引擎、库存核算 | 3-4 周 |
| Phase 4 | 完成风控、争议和后台运营 | Phase 3 输出 | 信用、风控规则、争议工单、运营后台 | 2-3 周 |
| Phase 5 | 压测、试运行与灰度上线 | 全量功能 | Pilot 版本、压测报告、上线清单 | 2 周 |

## 3. Phase 0：边界冻结

### 3.1 目标

- 明确一期只做 `MVP-Lite`
- 确认主底座为 `new-api-main`
- 冻结核心实体与账务模型

### 3.2 详细步骤

1. 以本目录文档作为启动版本
2. 在正式实施仓中创建架构 ADR 目录
3. 评审并冻结以下实体：
   - `seller`
   - `seller_key`
   - `listing`
   - `listing_sku`
   - `buyer_asset`
   - `entitlement`
   - `order`
   - `order_item`
   - `usage_ledger`
   - `settlement_ledger`
   - `dispute_case`
4. 约束一期不做的范围：
   - 自由竞价
   - 保险服务
   - 自动仲裁
   - 卖家自助提现
   - 多币种
5. 明确一期支持的卖家接入模式：
   - 平台白名单卖家
   - 人工审核
   - 平台托管密钥

### 3.3 输出

- 主数据模型
- API 边界草案
- 页面清单
- 一期范围清单

### 3.4 验收标准

- 不再对主底座反复切换
- 不再对核心账务模型反复摇摆

## 4. Phase 1：网关与卖家供给基础

### 4.1 目标

在 `new-api-main` 现有网关能力上，新增卖家供给层和密钥托管层。

### 4.2 详细步骤

1. 扩展 `channel` 概念，区分：
   - 平台自有渠道
   - 卖家托管渠道
2. 新增 `seller` 与 `seller_key_pool`
3. 建立 Seller Key Vault 抽象：
   - 加密存储
   - 使用审计
   - 轮换状态
4. 建立卖家密钥健康任务：
   - 可用性检测
   - 模型能力检测
   - 失败率统计
5. 为 `listing` 预留与 `seller_key_pool` 的映射
6. 为 Relay 增加“面向买方权益”的选路入口

### 4.3 输出

- 卖家域表结构
- 密钥托管能力
- 健康检测任务
- 基础路由扩展

### 4.4 验收标准

- 卖家密钥不会暴露到客户端
- 系统可以区分不同卖家的供给池

## 5. Phase 2：市场、下单、支付、权益

### 5.1 目标

跑通“浏览商品 -> 下单支付 -> 获得权益 -> 发起调用”的最小闭环。

### 5.2 详细步骤

1. 新增商品域：
   - 商品主表
   - SKU
   - 保障等级
   - 审核状态
2. 新增买方门户页面：
   - 商品列表
   - 商品详情
   - 我的资产
3. 建立订单域：
   - 购物车
   - 订单
   - 订单明细
   - 支付回调
4. 将支付成功后的落地对象从“直接加用户额度”改为：
   - 生成买方权益
   - 生成待分配 entitlement
5. 为 entitlement 设计扣减规则：
   - 同模型优先
   - 先到期先用
   - 同商品 FIFO

### 5.3 输出

- 商品页面
- 下单/支付闭环
- 买方权益账户

### 5.4 验收标准

- 用户购买后不拿到卖家 key
- 用户能够通过平台统一 Key 成功消费已购权益

## 6. Phase 3：计费、库存、结算

### 6.1 目标

把交易平台真正跑起来，确保“卖了多少、用了多少、该结多少”三者可核对。

### 6.2 详细步骤

1. 建立三本账：
   - `inventory_ledger`
   - `entitlement_ledger`
   - `usage_ledger`
2. Relay 调用链新增：
   - 请求预扣
   - 上游实际 usage 回写
   - 失败退款或修正
3. 为异步任务建立统一事件：
   - `order.paid`
   - `entitlement.created`
   - `usage.recorded`
   - `settlement.pending`
4. 新增结算引擎：
   - T+1 出账
   - 保障期冻结
   - 争议中冻结
   - 可结算余额更新
5. 新增库存同步任务：
   - 卖家可售库存刷新
   - 库存不足自动停售

### 6.3 输出

- 三本账体系
- 结算引擎
- 库存核算能力

### 6.4 验收标准

- 任意一笔请求能追到来源订单、权益、卖家供给和结算余额

## 7. Phase 4：风控、争议与运营后台

### 7.1 目标

让平台具备真实运营能力。

### 7.2 详细步骤

1. 新增卖家信用模型
2. 新增异常检测规则：
   - 低价异常
   - 高频失败
   - 异常登录
   - 刷单嫌疑
3. 新增争议工单域：
   - 工单创建
   - 证据上传
   - SLA
   - 裁决结果
4. 新增后台页面：
   - 卖家审核
   - 商品审核
   - 争议处理
   - 结算管理
   - 风控告警

### 7.3 输出

- 风控规则
- 争议工作台
- 平台运营后台

### 7.4 验收标准

- 平台管理员可独立处理交易问题
- 结算冻结与争议处理状态能够联动

## 8. Phase 5：试运行与灰度上线

### 8.1 目标

在白名单卖家和白名单买家范围内完成 Pilot。

### 8.2 详细步骤

1. 建立测试数据集：
   - 真实卖家
   - 模拟多类商品
   - 多种失败场景
2. 运行压测：
   - 1000 TPS 目标压测
   - 渠道失败注入
   - 结算一致性回放
3. 运行灰度：
   - 先平台自营商品
   - 再白名单卖家
   - 最后开放申请制卖家
4. 建立上线清单：
   - 监控项
   - 回滚项
   - 告警责任人

### 8.3 输出

- Pilot 报告
- 上线清单
- 回滚预案

### 8.4 验收标准

- Pilot 内账实一致
- 主要故障可以被监控和恢复

## 9. 里程碑设计

### 里程碑 1：主底座和领域边界确定

判定标准：

- 表设计和接口边界通过评审

### 里程碑 2：交易最小闭环打通

判定标准：

- 用户可完成浏览、购买、调用

### 里程碑 3：账务与结算跑通

判定标准：

- 账务链路可追溯且可核对

### 里程碑 4：运营后台与争议机制成形

判定标准：

- 平台具备日常运营能力

### 里程碑 5：Pilot 上线

判定标准：

- 白名单试运行稳定

## 10. 现实工期建议

### 10.1 MVP-Lite

- 6-8 周
- 适合验证闭环

### 10.2 Pilot MVP

- 12-16 周
- 适合带白名单卖家试运营

### 10.3 Marketplace V1

- 16-24 周
- 适合对外开放多卖家

## 11. 验证与测试策略

每个 Phase 都需要至少四类验证：

1. API 回归
2. 请求链路与流式回归
3. 账务一致性校验
4. 管理后台流程回归

特别要做的专项验证：

- 卖家密钥失效后的自动隔离
- 并发请求下 entitlement 扣减一致性
- 支付回调幂等
- 争议冻结期间结算阻断

## 12. 进入实施前的最终建议

如果只能立刻做一件事，建议先做：

`Phase 0 + Phase 1 的表设计和 Relay 扩展接口设计`

因为这决定后面所有业务是否能稳住。
