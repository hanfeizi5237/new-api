# 05-异常处理方案

## 1. 风险识别总表

| 风险类别 | 风险描述 | 概率 | 影响 | 风险等级 |
| --- | --- | --- | --- | --- |
| 合规风险 | 上游供应商不允许 resale 或 key delegation | 中高 | 很高 | 高 |
| 账务风险 | 预扣、实扣、退款修正不一致 | 中 | 很高 | 高 |
| 供给风险 | 卖家上架额度与真实可用额度不一致 | 高 | 高 | 高 |
| 工程风险 | 贸然从错误底座开工导致返工 | 中 | 高 | 高 |
| 性能风险 | 高并发下 entitlement 与 ledger 竞争冲突 | 中 | 高 | 高 |
| 支付风险 | 回调重复、篡改或丢失导致订单状态异常 | 中 | 高 | 高 |
| 运维风险 | 渠道集体失效但平台无隔离/告警 | 中 | 高 | 高 |
| 交易风险 | 恶意卖家、刷单、异常低价扰乱市场 | 中高 | 中高 | 高 |
| 产品风险 | 一期范围过大导致交付失败 | 高 | 高 | 高 |
| 数据风险 | 卖家密钥泄露或后台敏感字段明文暴露 | 低中 | 很高 | 高 |

## 2. 高风险预防措施

### 2.1 合规风险预防

- 一期只接入经过审核的卖家和白名单供应商
- 不对外暴露卖家原始 key
- 卖家协议中明确供给真实性和可用性责任
- 上线前逐项审查供应商条款和区域限制

### 2.2 账务风险预防

- 三本账分离
- 任何账务写入必须有幂等键
- 订单、权益、用量、结算必须可回放
- 对关键流水建立审计日志

### 2.3 供给风险预防

- 上架前做 key 有效性验证
- 周期性库存刷新
- 连续失败自动停售
- 对高风险卖家设置更短的保障期和更高保证金

### 2.4 工程风险预防

- 先做领域模型评审，不先写业务代码
- 明确一期不做的范围
- 以 `new-api-main` 为主底座，不再摇摆

## 3. 应对策略

| 风险 | 应对方案 | 责任方 | 触发条件 |
| --- | --- | --- | --- |
| 卖家 key 大面积失效 | 自动停售 + 通知运营 + 冻结新单 | 后端 + 运维 | 健康检测失败率超阈值 |
| 账务异常 | 冻结结算批次 + 触发回放程序 | 后端 | 对账不平 |
| 支付回调重复 | 订单幂等锁 + 状态机校验 | 后端 | 同一订单重复回调 |
| 争议爆发 | 冻结相关订单/卖家结算 | 运营 | 同卖家争议率升高 |
| 高并发锁冲突 | 降级到串行队列或热点分片 | 后端 | ledger 冲突率升高 |
| 平台网关异常 | 限流 + 熔断 + 切换备用渠道 | 运维 | 错误率升高 |
| 卖家恶意低价 | 风控标记 + 人工审核恢复 | 运营 | 价格低于阈值 |

## 4. 回溯与修复策略

### 4.1 什么情况下必须回溯

- 账务对不上
- 订单状态错乱
- entitlement 余额为负
- 可结算金额与 usage 明显不一致

### 4.2 如何回溯

按以下顺序回溯：

1. `order`
2. `order_item`
3. `buyer_asset / entitlement`
4. `usage_ledger`
5. `settlement_ledger`

核心原则：

- 所有修复通过补偿流水完成
- 不直接改历史事实记录

### 4.3 回溯后的恢复策略

- 冻结相关卖家/买家新交易
- 完成补偿后再解冻
- 记录事故单和根因

## 5. 监控指标

### 5.1 网关指标

- QPS
- 成功率
- P95 / P99
- 渠道失败率
- 自动切换次数

### 5.2 交易指标

- 商品上架通过率
- 下单成功率
- 支付回调成功率
- entitlement 创建成功率

### 5.3 账务指标

- 预扣与实扣偏差率
- 日结算差异笔数
- 冻结结算笔数
- 退款修正笔数

### 5.4 风控指标

- 卖家异常率
- 争议率
- 黑名单命中数
- 异常登录告警数

## 6. 告警阈值建议

| 指标 | 阈值 | 级别 |
| --- | --- | --- |
| 网关失败率 | > 5% | P1 |
| P99 延迟 | > 10s | P1 |
| 渠道连续失败 | >= 3 次 | P1 |
| 对账差异订单 | >= 1 单 | P1 |
| 同卖家争议率 | > 10% | P1 |
| 卖家 key 池不可用比例 | > 30% | P1 |
| 支付回调失败率 | > 2% | P2 |

## 7. 上线前必须通过的异常场景

### 7.1 交易链路

- 用户支付成功但回调延迟
- 用户支付成功但权益创建失败
- 用户退款后权益已被部分消耗

### 7.2 网关链路

- 上游超时
- 上游 429
- SSE 中途断流
- 路由切换后重复扣费

### 7.3 账务链路

- 并发扣减同一 entitlement
- 失败请求未退回预扣额度
- 结算批次中途失败

### 7.4 风控链路

- 低价商品自动拦截
- 高争议卖家自动冻结
- 异地登录告警

## 8. 一期的风险收缩建议

为了确保可落地，一期建议主动降低以下风险：

- 不开放完全自由卖家市场
- 不支持复杂竞价和组合包
- 不支持卖家自助提现
- 不支持完全自动裁决

这些控制会让平台在前两个月更稳。

## 9. 责任划分建议

### 后端负责

- 账务正确性
- 接口幂等
- 渠道健康与切换

### 前端负责

- 状态展示准确
- 风险提示透出
- 争议流程可用性

### 运维负责

- 监控与告警
- 备份与恢复
- 灰度发布

### 运营负责

- 卖家审核
- 商品审核
- 争议处理
- 信用分策略调整

## 10. 结论

本项目最大风险不是代码写不出来，而是：

- 范围过大
- 账务不清
- 合规没先审

所以异常处理方案的核心思想是：

- 先防扩散
- 再做补偿
- 保留事实流水
- 不直接手改历史账
